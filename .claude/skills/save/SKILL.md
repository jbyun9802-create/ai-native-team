# 작업 기록 저장 (프레임워크 기반)

팀원이 방금 한 AI 대화에서 핵심 내용을 **사고 프레임워크에 맞춰** 기록으로 저장합니다.

## 입력

사용자가 `/save` 뒤에 텍스트를 입력하면 그것을 기록 주제의 힌트로 사용합니다: $ARGUMENTS

## 실행 순서

### 1단계: 역할 확인
사용자에게 어떤 역할로 기록할지 물어봅니다:
- `pm` — PM (프로덕트 매니지먼트)
- `engineering` — 엔지니어링
- `design` — 디자인
- `uxr` — UX 리서치
- `gtm` — GTM (Go-To-Market)

### 2단계: 프레임워크 로딩 + 섹션 매칭

해당 역할의 사고 프레임워크 파일을 읽습니다:
- `pm` → `docs/shared/product/specs/frameworks/pm-thinking.md`
- `engineering` → `docs/shared/product/specs/frameworks/engineering-thinking.md`
- `design` → `docs/shared/product/specs/frameworks/design-thinking.md`
- `uxr` → `docs/shared/product/specs/frameworks/uxr-thinking.md`
- `gtm` → `docs/shared/product/specs/frameworks/growth-thinking.md`

> **환경 분기**: 웹(BYOK) 환경에서는 이 파일이 이미 system prompt에 `=== {framework_name}.md ===` 블록으로 주입되어 있습니다. Read 도구를 호출하지 말고 주입된 블록을 그대로 사용하세요. Claude Code 환경에서는 Read 도구로 직접 로딩하세요.

프레임워크를 읽은 후, 이 대화 전체를 분석하여:
1. 프레임워크의 **어떤 섹션**(번호 + 제목)이 이 대화와 관련되는지 파악합니다 (1~3개)
2. 매칭 근거를 간단히 정리합니다

**매칭이 안 되는 경우**: 대화 내용이 어떤 프레임워크 섹션에도 명확히 매칭되지 않으면, 사용자에게 "이 대화는 프레임워크의 특정 섹션에 매칭되지 않아 기본 포맷으로 저장합니다"라고 알리고 **6단계(Generic 저장)**로 건너뜁니다.

### 3단계: 사고 체크

매칭된 각 섹션의 **"사고 단계"**에 있는 Step들을 하나씩 체크합니다.
대화에서 해당 Step이 다뤄졌는지를 다음 3가지로 분류합니다:

| 상태 | 의미 |
|------|------|
| **Covered** | 대화에서 충분히 다뤄짐 |
| **Partially** | 언급은 됐지만 깊이가 부족 |
| **Not covered** | 다뤄지지 않음 |

체크 결과를 사용자에게 **표 형태**로 보여줍니다:

```
## 사고 체크: {섹션명}

| Step | 상태 | 요약 |
|------|------|------|
| Step 1: 문제가 진짜인가? | Covered | 유저 세그먼트와 빈도를 데이터로 확인 |
| Step 2: 문제의 구조는? | Not covered | — |
| Step 3: 풀 가치가 있는가? | Partially | 우회 방법은 논의했으나 시장 관점 부족 |
```

그 아래에 해당 섹션의 **"반드시 사람이 확인할 것"** 항목도 함께 보여줍니다:

```
### 사람이 확인해야 할 것
- [ ] "이 문제가 진짜인지"의 최종 판단
- [ ] 유저의 감정적 맥락
```

### 4단계: 보충 질문

Not covered 또는 Partially인 Step이 있으면, 프레임워크의 해당 Step 하위 bullet에서 **핵심 질문을 최대 3~5개** 뽑아 사용자에게 물어봅니다.

질문 우선순위:
1. "반드시 사람이 확인할 것"에 해당하는 항목
2. Not covered인 Step의 핵심 sub-bullet
3. Partially인 Step에서 빠진 부분

**질문 예시** (PM 문제 대하기 섹션):
> 빠진 부분이 있어서 몇 가지 여쭤볼게요:
> 1. 이 문제의 근본 원인 vs 증상을 구분하셨나요? 구분하셨다면 어떻게?
> 2. 안 풀면 어떤 일이 벌어지나요?
> 3. "이 문제가 진짜인지"에 대한 최종 판단은 어떤가요?

사용자가 **"skip"**, **"패스"**, **"해당 없음"**, **"그냥 저장해줘"** 등으로 응답하면 보충 질문을 건너뛰고 5단계로 진행합니다.

사용자가 답변하면, 해당 내용을 체크표에 반영하고 5단계로 진행합니다.

### 5단계: 내용 구조화 + 확인

대화 전체 + 보충 질문 답변을 종합하여 다음을 정리합니다:
- **제목**: 어떤 작업/주제에 대한 것인지 (간결하게)
- **목적**: 이 대화를 통해 하려 한 것
- **사고 단계별 내용**: 프레임워크 Step에 맞춰 구조화
- **결정사항**: 내린 결정이 있다면 무엇을, 왜 그렇게 결정했는지
- **인사이트**: 새로 알게 된 것, 생각이 바뀐 부분
사용자에게 정리한 내용을 보여주고 수정할 부분이 있는지 확인합니다.

### 5.5단계: 열린 질문 자동 감지

이 대화 전체를 분석하여 **아직 답이 나지 않은 질문, 보류된 결정, 다음에 생각해봐야 할 것**을 자동으로 감지합니다.

감지 소스:
- 대화에서 제기됐지만 결론이 안 난 질문
- "다음에", "나중에", "확인 필요" 등으로 미룬 것
- 다른 팀원의 피드백/확인이 필요하다고 언급된 것
- "반드시 사람이 확인할 것" 중 아직 확인 안 된 것
- 가정으로 남아있는 것 ("~일 것이다", "아마")

감지 결과를 사용자에게 보여줍니다:

```
## 열린 질문 (자동 감지)

이 대화에서 아직 열린 질문을 발견했습니다:

1. STOP 룰이 실제로 Claude Code를 멈추게 하는지 테스트 필요
2. 아롬님 UXR 방법론 피드백 후 재설계 필요
3. ...

맞나요? 수정하거나 추가할 것이 있으면 알려주세요.
(없으면 "확인" / 열린 질문이 없다면 "없음")
```

사용자가 확인/수정하면 다음 두 곳에 반영합니다:
1. **frontmatter**: `open_thinking: ["질문1", "질문2"]` 필드 추가
2. **본문**: `## 열린 질문` 섹션에 작성

사용자가 "없음"이라고 하면:
- frontmatter에 `open_thinking` 필드를 넣지 않음
- 본문에 `## 열린 질문` 섹션은 "없음"으로 작성

### 6단계: 파일 생성

오늘 날짜를 기반으로 파일을 생성합니다:
- 경로: `docs/roles/{역할}/traces/{오늘날짜}-{주제}.md`
- 날짜 형식: `YYYY-MM-DD`
- 주제는 한글로, 공백 대신 하이픈 사용

> **웹(BYOK) 환경**: 실제 파일 쓰기 도구가 없으므로 최종 마크다운을 ```` ```trace ```` 와 ```` ``` ```` 로 감싸서 출력하세요. 브라우저가 블록을 추출해서 GitHub Contents API로 커밋합니다.

#### 프레임워크 매칭된 경우의 파일 형식:

```markdown
---
date: {오늘 날짜}
type: trace
role: {선택한 역할}
topic: {제목 한 줄 요약}
project: ai-native-team
tags: [{핵심 키워드 3~5개, 쉼표로 구분}]
framework_sections: ["{매칭된 섹션명1}", "{매칭된 섹션명2}"]
thinking_coverage: "{covered+partially 수}/{전체 step 수} steps"
open_thinking: ["{열린 질문1}", "{열린 질문2}"]
---

# {제목}

## 날짜
{오늘 날짜}

## 목적
{정리한 목적}

## 사고 프레임워크: {섹션명}

### 사고 단계

| Step | 상태 | 내용 |
|------|------|------|
| Step 1: {step 제목} | Covered | {대화에서 다룬 근거/요약} |
| Step 2: {step 제목} | Covered | {근거/요약} |
| Step 3: {step 제목} | 보충 | {보충 질문에 대한 사용자 응답} |
| Step 4: {step 제목} | 해당 없음 | {사용자가 skip한 경우} |

(매칭된 섹션이 2개 이상이면 섹션별로 위 구조를 반복)

### 사람이 확인한 것
- [x] {확인 항목} — {사용자 확인 내용}
- [ ] {미확인 항목}

## 결정사항
{결정 내용과 이유. 없으면 "없음"}

## 인사이트
{새로 알게 된 것}

## 열린 질문
{5.5단계에서 확인된 열린 질문. 없으면 "없음"}

```

#### 프레임워크 매칭 안 된 경우 (Generic Fallback):


**tags 작성 규칙:**
- 대화에서 다뤄진 핵심 개념/도구/주제를 태그로 추출
- 나중에 "이 주제에 대해 논의한 적 있어?" 검색에 걸릴 수 있는 단어로 선정
- 예: [RAG, 메타데이터, Supabase, 검색구조]

### 7단계: 완료 메시지
저장된 파일 경로를 알려줍니다.
프레임워크 매칭이 된 경우, thinking_coverage도 함께 알려줍니다.
