# Product 폴더 구조 재설계 — 단계(Phase)별 정리

## 날짜
2026-03-15

## 목적
`docs/product/` 폴더의 구조를 비개발자·개발자 모두 이해하기 쉽게 재설계

## 핵심 내용
- product 폴더를 단계(Phase)별로 나눔: `planning/`, `design/`, `specs/`, `validation/`
- 코드는 `src/`에 분리, 문서만 `docs/product/`에 배치하기로 결정
- 폴더 사용법 가이드(`docs/product/README.md`) 생성
- 메타데이터 표준에 `phase` 필드 추가 (값: `planning` / `design` / `spec` / `validation`)
- `diagrams/`는 단계 무관 폴더로 유지
- 기존 `problem-analysis.md`를 `planning/` 하위로 이동

## 결정사항
1. **코드와 문서 분리**: 코드는 `src/`, 문서는 `docs/product/`에 배치. 왜? 개발자에게 익숙한 구조이면서 비개발자가 문서를 쉽게 찾을 수 있음
2. **단계(Phase)별 정리 채택**: 기능별·역할별 대신 단계별 구조 선택. 왜? 팀 전체 프로세스 흐름(기획→디자인→개발→검증)에 맞고 비개발자에게 직관적
3. **4개 단계 폴더**: planning(기획), design(디자인), specs(개발 명세), validation(검증)
4. **diagrams/ 유지**: 다이어그램은 특정 단계에 종속되지 않으므로 별도 유지

## 인사이트
- 태스크별·역할별보다 단계별 구조가 비개발자에게 더 직관적임
- 빈 폴더도 `.gitkeep`으로 미리 만들어 두면 팀원이 "여기에 넣으면 되겠구나" 하고 인지할 수 있음
- README 가이드에 "각 폴더에 뭘 넣는지" 표로 정리하면 온보딩이 쉬워짐

## 다른 팀원이 알아야 할 것
- `problem-analysis.md` 경로가 `docs/product/planning/`으로 변경됨
- 앞으로 product 관련 문서 작성 시 해당 단계 폴더에 넣고, 메타데이터에 `phase` 필드를 추가해야 함
- 폴더 사용법은 `docs/product/README.md` 참고
