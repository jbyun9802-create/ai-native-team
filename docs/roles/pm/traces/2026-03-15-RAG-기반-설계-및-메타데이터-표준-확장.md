# RAG 기반 설계 및 메타데이터 표준 확장

## 날짜
2026-03-15

## 목적
- AI-native team 프로젝트에 RAG를 처음부터 전제한 문서 구조를 만들기
- 미팅 원문 자동 저장 방식 결정
- 메타데이터 표준을 프로덕트 개발 전체 주기에 맞게 확장

## 핵심 내용
- RAG란 ①저장(Embedding) → ②검색(Retrieval) → ③답변생성(Generation) 3단계로 구성됨
- 현재 규모에서는 벡터 DB(Supabase) 없이 Claude Code의 파일 검색만으로 충분 (파일 기반 RAG)
- 문서가 수천 개 이상 쌓이면 Supabase 벡터 DB + MCP 서버로 전환
- Supabase vs Pinecone: 현재 규모에선 Supabase가 적합 (만능 창고 vs 검색 전용 도서관)
- 미팅 녹화는 Fireflies Pro (월 $10) 선택 — API/Webhook으로 Git 자동 저장 가능, 본인 미참석 미팅에도 봇 전송 가능
- Granola는 본인 참석 미팅만 가능해서 탈락
- 메타데이터 백필 완료 (18개 전체 파일)
- 새 type 4개 추가: `research`, `spec`, `role-overview`, `guide`
- 새 필드 추가: `status`(spec용), `method`(research용), `supersedes`(버전 관리), `phase`(product용)

## 결정사항
- 미팅 레코더: Fireflies Pro 사용 (월 $10, 1명만 구독)
- RAG 인터페이스: 슬랙봇이 아닌 Claude Code 채팅창에서 사용
- 벡터 DB: 지금은 불필요, 데이터 충분히 쌓이면 Supabase로 도입
- sprint, design, launch 등의 type은 기존 type + 태그 조합으로 충분하므로 추가하지 않음

## 인사이트
- 메타데이터를 잘 붙여놓으면 파일 기반으로도 RAG와 유사한 검색이 가능함
- RAG 도입의 핵심은 기술 세팅이 아니라 "데이터를 잘 쌓는 것"
- 원문은 가공하지 않고 통째로 저장하는 게 핵심 (뉘앙스, 반대 의견 등이 나중에 중요한 단서)
- `supersedes` 필드로 문서 버전 관리가 가능해짐

## 다른 팀원이 알아야 할 것
- 모든 .md 문서에 YAML 메타데이터가 필수가 됨 — metadata-standard.md 참고
- 미팅 후 채팅/트랜스크립트 원문을 `docs/shared/meetings/날짜-주제/` 폴더에 txt로 저장할 것
- Fireflies 연동이 세팅되면 자동 저장됨
