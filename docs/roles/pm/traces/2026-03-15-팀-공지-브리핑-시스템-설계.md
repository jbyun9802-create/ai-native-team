# 팀 공지 & 브리핑 시스템 설계

## 날짜
2026-03-15

## 목적
팀원 간 정보 공유 파이프라인 만들기 — trace 작업 중 다른 팀원에게 공유할 내용을 전달하고, 아침에 브리핑으로 확인할 수 있는 워크플로우 구축

## 핵심 내용
- `/save` 할 때 "다른 팀원이 알아야 할 것"이 있으면 공지를 자동 생성하는 흐름 추가
- 공지 파일은 `docs/shared/announcements/`에 저장, frontmatter로 `from`, `to`, `date`, `subject` 관리
- `to` 필드로 대상 지정: `all`(전체) 또는 특정 역할 목록 (예: `[engineering, design]`)
- `/briefing` 커맨드 신규 생성 — 역할 선택 후 나에게 온 공지 + 팀 최근 3일 활동 + 관련 사항 하이라이트

## 결정사항
- 공지 대상은 역할 단위로 지정. 이유: 현재 팀 구조에서 역할 = 사람이므로 별도 개인 지정 불필요
- 공지는 `/save` 흐름에 통합 (별도 커맨드 없이). 이유: 작업 기록을 남기는 시점이 공유 판단을 하기에 가장 자연스러운 타이밍

## 인사이트
- 파일 기반 공지 시스템이면 별도 인프라 없이 git으로 팀 커뮤니케이션 가능
- `/briefing`이 traces + announcements 두 소스를 합쳐서 보여주면 정보 누락 방지

## 다른 팀원이 알아야 할 것
- `/briefing` 커맨드가 생겼음 — 아침에 `/briefing` 치면 나에게 온 공지와 팀 현황을 한눈에 볼 수 있음
- `/save` 할 때 공유할 내용이 있으면 공지를 생성할지 물어봄 — 대상(전체 또는 특정 역할)을 선택 가능
