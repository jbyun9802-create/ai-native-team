# /save 커맨드 프레임워크 기반 강화

## 날짜
2026-04-04

## 목적
`/save` 실행 시 직무별 사고 프레임워크에 맞춰 사고 여부를 체크하고, 빠진 부분을 보충해서 구조화된 trace로 저장하도록 개선

## 사고 프레임워크: 문제를 대하는 방식

### 사고 단계

| Step | 상태 | 내용 |
|------|------|------|
| Step 1: 문제가 진짜인가? | Covered | 현재 /save가 무지성으로 저장되어 trace의 컨텍스트 의미가 사라지는 문제 확인 |
| Step 2: 문제의 구조는? | 보충 | 근본 원인이 이중적 — 프레임워크 미내재화 + 저장 구조가 사고를 강제하지 않음 |
| Step 3: 풀 가치가 있는가? | 보충 | 안 풀면 traces가 의미 없는 데이터로 쌓임. 풀면 나중에 의미 있는 컨텍스트와 사고 과정 데이터가 됨 |

## 사고 프레임워크: PRD를 쓰는 방식

### 사고 단계

| Step | 상태 | 내용 |
|------|------|------|
| Step 1: 프레스 릴리즈 | Covered | /save가 단순 저장이 아닌 "사고 습관화 도구"로 역할 확장 |
| Step 2: 성공 지표 | 보충 | traces가 의미 있는 컨텍스트와 사고 과정 데이터가 되는 것이 성공 기준 |
| Step 3: 스코프 | Covered | 프레임워크 매칭 → 사고 체크 → 보충 질문(3~5개) → 구조화 저장. skip 가능, generic fallback 포함 |
| Step 4: 리스크 | 보충 | 핵심 리스크: 너무 복잡하고 오래 걸려서 안 쓰게 될 수 있음 → skip 옵션과 질문 수 제한으로 대응 |
| Step 5: 유저 시나리오 | Covered | 해피 패스 + 엣지케이스(skip, fallback, 다중 섹션 매칭) 설계 완료 |

### 사람이 확인한 것
- [x] 문제가 진짜인지 — 무지성 저장으로 컨텍스트 의미 소실 확인
- [ ] 유저 시나리오의 현실성 — 팀원들이 실제로 이 흐름을 자연스럽게 쓸지 (배포 후 확인 필요)
- [x] 스코프 적절성 — skip 가능 + 질문 제한으로 과도한 복잡성 방지

## 결정사항
- 프레임워크 매칭 → 사고 단계 체크 → 보충 질문 → 구조화 저장의 흐름으로 설계
- 보충 질문은 최대 3~5개로 제한, skip 가능 (과도한 복잡성 방지)
- 프레임워크에 안 맞는 대화는 기존 generic 포맷으로 fallback
- YAML에 framework_sections, thinking_coverage 메타데이터 추가

## 인사이트
- 근본 원인이 이중적: 프레임워크 미내재화 + 저장 구조가 사고를 강제하지 않음 → 커맨드가 "사고 습관화 도구" 역할까지 해야 함
- 핵심 리스크는 "너무 복잡해서 안 쓰게 됨" — skip 옵션과 질문 수 제한이 중요
