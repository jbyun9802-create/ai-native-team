# 다음주 우선순위 — 질문지→리포트→협업 룰 변환 엔진 보강

## 날짜
2026-04-05

## 목적
- 프로덕트의 핵심 엔진(질문지 → 리포트 → {이름}.md + 팀 CLAUDE.md 변환)이 현재 sloppy하게 정의되어 있고 eval 프로세스가 전무한 현실을 직시
- "팀이 이걸 도입했을 때 실제로 도움이 되는가?" — 이 질문에 답할 수 있는 품질 기준과 검증 체계가 없음
- 다음주 최우선 과제로 설정하고, 필요한 피드백(아롬님, 우석님) 구조화

## 핵심 내용

### 왜 이것이 제일 중요한가

이 엔진이 프로덕트 존재 이유 자체다:

```
88문항 응답 → 6축 유형 결정 → Claude API 호출
  → 개인: {이름}-report.md + {이름}.md
  → 팀: team-report.md + CLAUDE.md (+ 보강된 {이름}.md × 전원)
```

이 체인의 최종 산출물이 **팀의 .claude/ 폴더에 들어가서 매일 작동하는 룰**이 된다.
질문지가 정확해야 → 리포트가 의미있고 → {이름}.md가 실제로 사고를 강제하고 → 팀 CLAUDE.md가 실제로 협업을 개선한다.
**이 체인 어디가 무너져도 프로덕트 전체가 무너진다.**

그런데 지금, 이 체인의 최종 산출물에 대한 **"잘 된 것"의 정의가 아예 없다.**

---

### 무엇이 sloppy한가 — 전체 산출물 체계의 품질 정의 부재

#### 1. {이름}.md — 개인 룰 파일의 품질이 뭔지 모름

robin.md 예시 하나가 있다 (assessment-questions.md Part 2). 하지만:

- **STOP 룰**: "멈추고 물어보세요" — Claude Code가 실제로 이 지시를 따라 멈추는지 테스트한 적 없음. 멈추는 것 자체가 목적이 아니라, 멈춤으로써 사용자가 **더 나은 판단**을 하게 되는지가 핵심인데, 이걸 어떻게 판단하지?
- **THINK 사고 사다리**: Step 1~4로 단계를 밟게 하는데, 이 사다리를 밟은 결과물이 안 밟은 결과물보다 **실제로 더 나은지** 어떻게 알지?
- **함정 패턴**: "같은 문서를 30분 이상 수정하고 있을 때" 감지한다는데, Claude Code가 시간을 추적하나? 실현 가능한가?
- **결과물 전달 룰**: "우석에게 보낼 때 상단 3줄 요약 필수" — 이 룰이 있으면 실제로 우석이 작업물을 더 잘 이해하게 되나?

→ robin.md가 "좋은" {이름}.md인지 "나쁜" {이름}.md인지 **판단할 기준 자체가 없다**.

#### 2. 팀 CLAUDE.md — 팀 전체 룰의 품질이 뭔지 모름

현재 CLAUDE.md 구조:
```
사용자 확인 → {이름}.md 참조 → 팀 공통 룰 → 결과물 전달 룰 → 사후 성찰
```

문제:
- **"사용자 확인 → {이름}.md 참조"**: Claude Code가 세션마다 "누구세요?" 묻고 해당 .md를 로드하는 흐름이 실제로 자연스럽게 작동하는지 미검증
- **팀 공통 룰 3개**: "최종 판단은 사람이 먼저" / "reasoning 포함" / "기각한 대안 포함" — 이 3개가 왜 이 3개인지? 우리 팀 유형 조합에서 도출된 건지, 범용적으로 좋은 룰인지?
- **결과물 전달 룰**: PM→Design, PM→Eng, Design→Eng, UXR→PM 등 모든 페어에 대해 "필수 포함 정보"가 정의됨. 그런데:
  - 이 룰들이 유형 점수에서 자동 도출되는 건지, 사람이 수작업으로 쓴 건지 경계가 불명확
  - "Robin은 Explicit, 우석은 Implicit → 문서 길이 갭" — 이 해석이 맞는지 검증 안됨
  - 실제로 이 전달 룰을 따랐을 때 핸드오프가 개선되는지 측정 방법 없음
- **팀이 이걸 도입했을 때의 가치**: CLAUDE.md + {이름}.md를 .claude/에 넣고 git push한 순간부터, 팀원 모두의 Claude Code 세션이 이 룰을 따르게 됨. **"따른 결과 뭐가 좋아졌는데?"**에 답할 수 없음

#### 3. 개인 리포트 — "MBTI급 깊이"의 실체가 없음

리포트 8개 섹션(유형 개요 → 축별 분석 → 리터러시 → 직무 가이드 → 위험 패턴 → 성장 추천 → 협업 → 룰 프리뷰)이 정의됨. 그런데:
- "구체적이고 행동 가능한 추천" — "구체적"과 "행동 가능한"의 기준이 뭔데?
- "이 유형의 PM이 가장 빛나는 순간" — 이게 사용자가 읽었을 때 "아 맞아 이거 나야"라고 느끼게 하는 건 어떤 속성이지?
- MBTI를 벤치마크로 잡았으면, MBTI 리포트의 어떤 특성이 "좋은 리포트"인지 분석했어야 함. 안 했음.

#### 4. Claude API 프롬프트 — 입력 같은데 출력 품질이 다를 것

- 같은 유형(SGTC-EF PM Lv4)으로 3번 생성하면, 3개의 {이름}.md가 다른 품질/깊이로 나올 것
- 프롬프트 템플릿은 있지만 (Section 7.5-7.7), **좋은 출력과 나쁜 출력의 예시가 없음**
- "좋은 {이름}.md"의 정의가 없으니 프롬프트 개선 방향도 정할 수 없음

#### 5. 60쌍 문항 — 변별력 미검증

- 10쌍/축인데, 어떤 쌍이 실제로 유형을 잘 구분하는지 모름
- 중립 응답(3점) 반복 시 → 50% 근처 → 유형 문자가 랜덤 플립 → 무의미한 룰 생성
- 항목 분석(item analysis), 신뢰도 검증 전무

#### 6. 팀 분석 충돌 임계값

- "축 차이 30%면 충돌" — 축 1(판단 기점 S/A)의 30%와 축 5(소통 스타일 E/I)의 30%가 같은 심각도인가?
- 어떤 마찰을 팀 리포트에서 먼저 언급할지 우선순위 없음
- 축별 가중치가 필요한데 정의 안됨

---

### eval 프로세스가 왜 없었는가

- **구조 설계에 매몰됨**: 6축, 60쌍, 4개 Claude API 호출, 7개 리포트 섹션... "형식"에 집중하느라 "이게 실제로 작동하나?"를 건너뜀
- **프로토타입 착각**: 프로토타입이 돌아가니까 "됐다"고 느낌. 그런데 eval 없는 프로토타입은 데모일 뿐
- **robin.md 예시 착각**: 예시가 하나 있으니까 "정의됐다"고 느낌. 그런데 그 예시가 좋은 건지 나쁜 건지 모르면 정의된 게 아님

---

### 다음주 할 것 — 품질 정의 + eval 프로세스 설계

**1순위: "좋은 룰"이란 무엇인가 — 품질 기준 정의**

| 산출물 | 정의해야 할 것 |
|--------|--------------|
| {이름}.md | STOP 룰이 "좋다"는 기준 / THINK 사다리가 "효과있다"는 기준 / 함정 패턴이 "실현가능하다"는 기준 |
| CLAUDE.md | 팀 공통 룰이 "이 팀에 맞다"는 기준 / 전달 룰이 "핸드오프를 개선한다"는 기준 |
| 개인 리포트 | "읽은 사람이 자기를 인식한다"의 기준 / "행동 가능한 추천"의 기준 |
| 팀 리포트 | "마찰 예측이 맞다"의 기준 |

**2순위: 실행 테스트**
- {이름}.md를 .claude/에 넣고 실제 Claude Code 세션에서 STOP이 작동하는지 테스트
- CLAUDE.md의 "사용자 확인 → {이름}.md 참조" 흐름이 자연스럽게 돌아가는지 테스트
- 결과물 전달 룰이 적용된 상태에서 실제 핸드오프 해보기

**3순위: 내부 파일럿 (4인)**
- 4인 팀이 직접 테스트 완주 → 60쌍 응답 수집
- 생성된 리포트와 룰을 각자 읽고 "이거 나한테 맞아?" 평가
- 2주간 실사용 후 "실제로 도움이 됐나?" 검증

**아롬님과 논의할 것:**
- {이름}.md/CLAUDE.md 품질 기준 정의에 UXR 방법론 적용
- "도움이 된다"를 어떻게 측정할지 — 서베이 설계? 행동 관찰?
- 리포트 정확도 검증 방법

### 우석님 피드백 — UX friction 3개 영역

우석님: "이 유저 플로우가 맞는지 한번 천천히 생각해보고 가자"
→ 프로토타입 유저 friction이 큼. 아래 3개 영역으로 구조화.

**영역 1: 질문지→룰 변환의 채널 전환**

| 현재 | 문제 | 필요한 것 |
|------|------|----------|
| 웹에서 테스트 → zip 다운로드 → .claude/에 넣기 → git push | 웹→로컬→Git으로 3단 채널 전환 | 이 friction을 줄이거나 없앨 방법 |
| 질문지 리포트가 claude 규칙으로 변환되는 과정이 블랙박스 | 유저가 "이게 왜 이 룰이 됐지?" 이해 못함 | 변환 과정의 투명성 |

→ "두 개의 채널을 쓰는게 최선인가?"

**영역 2: 룰 적용 후 "그래서 뭘 어떻게 하라고?"**

| 현재 | 문제 | 필요한 것 |
|------|------|----------|
| .claude/에 파일이 들어간 상태 | 유저가 다음 행동을 모름 | 온보딩 플로우 |
| /save 커맨드 기반 사고 기록 | 유저 교육 부재 | /save 사용 가이드 & 동기 부여 |
| 직무별 사고 프레임워크 기반 작동 | 프레임워크의 존재 자체를 모를 수 있음 | 프레임워크 발견 경로 |

→ git으로 claude 룰이 import된 이후에 "그래서 뭘 어떻게 하라고?"를 해결해야 함

**영역 3: 공동 멘탈 모델의 시각화/접근성**

| 현재 | 문제 | 필요한 것 |
|------|------|----------|
| git 안에 .md 파일로 존재 | 시각화 없음, 검색 어려움 | 멘탈 모델 시각화 + 검색 UX |
| brain.py TUI 계획 (PRD-2) | TUI 실행 자체에 friction | TUI가 최선인지 재검토 |

→ "TUI가 최선인가?"

**우석님 기여**: "원 소스에서 원 브레인으로" 캐치프레이즈. 머리에 꽂히는 워딩으로 프레이밍 능력이 대단하심. 솔루션 방향은 얼라인됨. 실행 디테일에서 UX 전문가 조언 필요.

### 아롬님에게 필요한 것

- 위 UX friction 3개 영역 전체에 대한 UX 전문가 관점
- {이름}.md + CLAUDE.md 품질 eval에서 UXR 방법론 조언
  - "이 룰이 도움이 된다"를 어떻게 측정하는가?
  - 리포트 정확도를 어떤 서베이/인터뷰로 검증하는가?
- 유저 온보딩 플로우 — 룰 적용 후 "다음에 뭘 하라고?"를 어떻게 전달하는가

## 결정사항

| 결정 | 이유 |
|------|------|
| 다음주 1순위: 변환 엔진의 품질 정의 + eval 설계 | 최종 산출물({이름}.md + CLAUDE.md)의 "좋은 것"이 뭔지 모르면 프로덕트 자체가 성립하지 않음 |
| {이름}.md + CLAUDE.md 둘 다 품질 기준 필요 | 개인 룰만이 아니라 팀 전체 룰(CLAUDE.md → {이름}.md 참조 구조)의 작동 여부가 핵심 |
| eval을 아롬님과 병렬 진행 | 품질 기준 정의(PM) + 검증 방법론(UXR)을 동시에 잡아야 |
| 우석님 UX friction을 3개 영역으로 구조화 | 산발적 이슈를 체계적으로 논의할 수 있게 |

## 인사이트

- **"형식이 있다 ≠ 잘 정의되어있다"**: robin.md 예시가 있고, CLAUDE.md 예시가 있고, 6축 60쌍이 있으니까 "구조 잘 잡혀있네"라고 착각했다. 그런데 최종 산출물의 품질 기준이 없으면 구조도 무의미.
- **"팀이 도입했을 때 도움이 되는가?"가 유일한 질문**: 60쌍이 몇 쌍인지, 리포트가 몇 섹션인지는 중요하지 않다. CLAUDE.md와 {이름}.md가 .claude/에 들어간 후 팀의 협업이 실제로 개선되는가? 이 질문에 지금은 답할 수 없다.
- **eval 없는 프로토타입은 데모**: 돌아가는 프로토타입에 안심하면 안 됨. 품질 기준을 세우고 측정해야 프로덕트가 됨.
- **우석님의 "천천히 생각해보자"**: 이 피드백이 정확히 핵심을 짚음. 빠르게 만드는 데 집중하느라 "이게 맞나?"를 건너뜀.
