## 사고 프레임워크: PRD를 쓰는 방식

### 사고 단계

| Step | 상태 | 내용 |
|------|------|------|
| Step 1: Working Backwards | Covered | "/save로 열린 질문 캡처 → TUI에서 t키로 미해결 질문 보기" |
| Step 2: 성공 지표 | Covered | /save에서 열린질문 섹션 비율 50%+, 의미있는 투두 3~7개 추출 |
| Step 3: 스코프 | Covered | 최소 범위 6개 항목 + "안 하는 것" 명시 |
| Step 4: 리스크 | Covered | 3개 가정(forward-looking 데이터 충분, 우선순위 3단계 유용, Haiku 품질) + 검증 방법 |
| Step 5: 유저 시나리오 | Partially | 해피 패스(t키 토글, 클릭) 다룸. API 실패 시 fallback 구현. 엣지케이스(trace 0개) 미상세 |

## 결정사항

| 결정 | 이유 |
|------|------|
| 데이터 소스: trace 자동 추출 (별도 파일 X) | trace가 source of truth인 철학과 일치 |
| UI: ModeButton 클릭 가능 위젯 + t키 토글 | 사용자 요청 — 버튼이 눈에 보여야 함 |
| 우선순위 3단계: ⚡blocking → ◆important → ○exploratory | 바 그래프 아닌 우선순위 리스트 형태로 (사용자 요청) |
| /save 열린 질문: Claude 자동 감지 → 사용자 확인 | "그냥 물어보는 게 아니라 자동 판단해서 보여줘야" (사용자 피드백) |
| trace 구조 보강 필수 | "input이 보장되지 않으면 output도 불안정" — 1층과 2층 동시 해결 |
| API 호출 비동기화 | 동기 호출로 UI 블로킹 → @work(thread=True) + "분석 중..." 로딩 표시 |
| 15초 polling 자동 갱신 | 새 trace 생성 시 자동 반영. glob+stat만 체크하니 부하 무시 가능 |

## 인사이트

- **"input이 보장되지 않으면 output도 불안정"**: TUI 기능만 만들면 안 되고 trace 구조(입력)를 먼저 고쳐야 의미있는 출력이 나옴. 2층 문제 구조를 §1 Step 2에서 발견
- **PM 프레임워크 적용이 설계 품질을 올렸음**: §1로 문제 2층 구조 발견, §2 Step 3 "안 하는 것"으로 스코프 조절, §3으로 리스크 명시. 사용자가 "pm framework 적용해서 다시 생각해"라고 요청한 것이 전환점
- **사용자 피드백이 설계를 날카롭게 함**: "그냥 물어보지 말고 자동 감지해서 보여줘야지" → /save 5.5단계 방식 개선. "바 그래프 아니라 우선순위" → 렌더링 방식 변경
