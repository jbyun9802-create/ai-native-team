# 레포 구조 재설계 및 웨비나/프로덕트 분리

## 날짜
2026-03-15

## 목적
팀 레포의 폴더 구조를 프로젝트 목적에 맞게 재설계하고, 웨비나와 프로덕트의 타겟/목적을 분리하며, 전체를 한국어화

## 핵심 내용
- 기존 `shared/roles` 구조가 애매해서, 목적 기반 구조로 전환 (`meetings/`, `product/`, `roles/`)
- README를 웨비나 섹션(목적: AI-native 팀이 어디까지 가능한지 시연, 타겟: AI 활용하려는 모든 현업인)과 프로덕트 섹션(목적: 우리 팀의 문제 해결)으로 분리
- 미팅 3건을 슬랙 히스토리 기반으로 구조화된 템플릿(날짜/참석자/안건/논의/결정/다음스텝)으로 재작성
- 문제 분석 문서를 한국어로 전환 (Main 1: 정보 손실, Main 2: 프로세스 부재, MVP: 판단이 붙은 핸드오프)
- 다이어그램 HTML(스윔레인 + 우선순위/MVP) 생성

## 결정사항
- **폴더 구조**: `docs/meetings/` + `docs/product/` + `docs/roles/` (shared 폴더 제거). 이유: docs/ 아래는 전부 팀 공유 자료이므로 `shared`라는 중간 계층이 불필요. 목적별로 나누는 것이 AI 에이전트가 탐색하기 쉬움
- **미팅 기록 형식**: 구조화된 템플릿 (날짜/참석자/안건/논의/결정/다음스텝). 이유: AI 에이전트가 파싱하기 좋고, 팀원이 빠르게 훑을 수 있음
- **역할별 traces 시스템**: 각 role 폴더에 `traces/` 하위 폴더로 AI 대화 기록 저장
- **미팅 원문 보관**: `shared/meetings/`에 raw-chat.txt, raw-transcript.txt 별도 보관
- **웨비나 vs 프로덕트 분리**: README에서 목적과 타겟을 명확히 구분

## 인사이트
- 이 레포 자체가 "판단이 붙은 핸드오프" MVP의 실험장 — 프로덕트를 만들면서 동시에 그 문제를 해결하는 메타적 구조
- 폴더 구조는 프로젝트의 목적을 반영해야 함. 기술적 분류(shared/roles)보다 목적 분류(meetings/product/roles)가 더 직관적

## 다른 팀원이 알아야 할 것
- 폴더 구조가 바뀌었으므로 기존 경로 참조 업데이트 필요
- 각 역할 폴더에 traces 기록 방식 가이드가 추가됨 (`TEMPLATE-trace.md` 참조)
- `/save` 커맨드로 AI 대화 기록을 자동 저장할 수 있음
