# 소스코드 레포 vs 협업 레포 — 프로덕트 레포 통합 전략

## 날짜
2026-04-05

## 목적

- TeamBrain이 생성한 CLAUDE.md 룰을 유저(프로덕트팀)가 도입할 때, **어떤 레포에 통합해야 하는가?**를 PM 프레임워크(§1, §2, §3, §7)에 매핑하여 사고 과정을 기록한다
- 이건 우리 팀 내부 구조 문제가 아니라 **프로덕트의 근본 설계 질문**이다
- 현재 PRD와 프로토타입에서 이 시나리오가 빠져 있음을 발견하고, 추가해야 할 내용을 정의한다

---

## 핵심 내용 — PM 프레임워크 기반 사고 흐름

### §1 Step 1: 문제가 진짜인가?

> **STOP**: 솔루션(레포 구조)으로 가기 전에 — 누가 어떤 문제를 겪는가?

**유저 시나리오를 따라가 보자:**

1. PM이 TeamBrain Assessment를 완료한다
2. 팀원별 CLAUDE.md 룰이 생성된다
3. 이 룰을 팀의 Git 레포에 넣어야 한다 (`.claude/` 디렉토리)
4. 팀의 Git 레포가 **소스코드 레포**인 경우 — PM에게 write 권한을 줘야 한다
5. PM이 소스코드 레포에 write 권한을 가지면 — CI/CD, config, .env 등을 건드릴 수 있다

**누가 겪는가?** 프로덕트팀 중 비개발자(PM, 디자이너, UXR). 소스코드 레포에 접근해야 하지만 코드를 건드리면 안 되는 사람들.

**얼마나 자주?** 매번. TeamBrain을 도입하는 모든 팀이 이 질문을 마주한다.

**얼마나 아픈가?** 팀 규모와 문화에 따라 다름. 4명 스타트업은 크게 문제 안 됨. 20명 이상이거나 규제 산업이면 "비개발자에게 코드 레포 권한"은 보안 정책 위반일 수 있음.

**데이터가 있는가, 가설인가?** 가설이다. 우리 팀(4명)에서는 아직 사고가 난 적 없다. 하지만 프로덕트가 다른 팀에 확산될 때를 대비해야 한다.

### §1 Step 2: 문제의 구조는?

> **THINK**: 근본 원인 vs 증상을 구분하자

- **증상**: "비개발자가 코드를 건드릴 수 있다"
- **근본 원인**: **GitHub는 경로별 write 제한을 지원하지 않는다**

이게 이 문제의 핵심이다. GitHub의 권한 모델을 정리하면:

| 메커니즘 | 할 수 있는 것 | 할 수 없는 것 |
|----------|-------------|-------------|
| **레포 권한** (Read/Write/Admin) | 레포 전체에 대한 접근 제어 | **경로별 제한 불가** — Write면 모든 경로에 쓸 수 있음 |
| **CODEOWNERS** | PR에서 특정 경로 변경 시 리뷰어 지정 | **write 자체를 차단하지 못함** — 리뷰 요구만 |
| **Branch Protection** | main 직접 push 차단, PR 필수화 | 경로별 구분 없음 — 모든 PR에 동일 적용 |
| **Push Rulesets** (경로 기반) | **경로별 push 차단 가능!** | **GitHub Team 플랜($4/user/월) 이상에서만** |

**결정적 발견**: "이 유저는 `docs/`만 쓸 수 있고 `src/`는 못 쓴다"를 GitHub에서 구현하려면 Team 플랜이 필요하다. Free 플랜에서는 불가능하다.

**연결된 문제들**:
- AI 에이전트 컨텍스트: 레포를 분리하면 Claude가 소스코드를 못 읽음
- 온보딩 플로우: 현재 PRD/프로토타입에서 이 분기를 다루지 않음
- /save 커맨드: 같은 레포 내 `docs/roles/{role}/traces/`에 쓰도록 설계됨

### §1 Step 3: 풀 가치가 있는가?

> **THINK**: 안 풀면 어떤 일이 벌어지는가?

- 프로덕트 퍼널 ③("Git 레포에 .claude/ 연결")에서 팀마다 "우리 소스코드 레포에 넣어도 되나?"라는 질문이 나옴
- 답이 없으면 팀이 스스로 판단해야 함 → 도입 허들 증가 → 전환율 하락
- 일부 팀은 보안 정책 때문에 도입 자체를 포기할 수 있음

**지금 유저는 어떻게 우회하고 있는가?** 우회할 방법이 없다. PRD에 이 시나리오가 없으니까.

### §2 Step 1: 이번에 반드시 달성해야 하는 한 가지 목표는?

> **AI 에이전트가 소스코드 + 사고 trace + CLAUDE.md 룰을 모두 읽을 수 있는 상태**를, 비개발자의 코드 접근 리스크 없이 만든다.

### §2 Step 2: 후보 평가 (RICE)

| 기준 | (A) 소스코드 레포에 그냥 통합 | (B) 별도 협업 레포 분리 | (C) 소스코드 레포 + 보호 장치 |
|------|--------------------------|---------------------|--------------------------|
| **Reach** | 높음 — 모든 팀 적용 가능 | 높음 | GitHub Team 플랜 팀만 완전 보호 |
| **Impact** | **AI 컨텍스트 최대** — 코드+trace+룰 통합 | AI 컨텍스트 단절 — Claude가 코드 못 읽음 | **AI 컨텍스트 최대 + 안전** |
| **Confidence** | 중간 — 권한 리스크 존재 | 중간 — 컨텍스트 손실의 영향 불확실 | **높음** — 단, GitHub 플랜에 의존 |
| **Effort** | 최소 — 아무것도 안 함 | 중간 — 2개 레포 세팅, 교차 참조 | 낮음 — CODEOWNERS + branch protection |

> **STOP**: (B)의 치명적 문제를 확인하자

**(B) 별도 레포가 안 되는 진짜 이유:**

TeamBrain의 핵심 가치는 "AI 에이전트가 팀의 사고 컨텍스트를 읽고 작업에 반영하는 것"이다.
- PM이 trace에 "우리가 X를 Y 때문에 결정했다"고 기록
- 엔지니어가 Claude에게 "X를 구현해줘"라고 요청
- Claude가 **같은 레포 안에서** trace를 읽고 Y의 맥락을 이해하며 구현

레포가 분리되면 이 체인이 끊긴다. 엔지니어의 Claude는 코드 레포만 보고, PM의 trace는 다른 레포에 있다. 교차 참조를 설정할 수는 있지만 ("다른 레포도 읽어라"), 자동 검색이 안 되고, 설정이 깨지기 쉽고, "zero configuration" 약속이 무너진다.

**이것이 전통적 레포 분리 논의와 다른 점**: AI 에이전트 컨텍스트라는 새로운 변수가 있다. 예전에는 "레포 분리하면 깔끔하고 권한 관리도 쉽다"가 정답이었지만, AI 에이전트 시대에는 **레포 분리 = 컨텍스트 단절 = 핵심 가치 훼손**이 된다.

### §2 Step 3: "안 하는 것" 명시

| 안 하는 것 | 이유 |
|-----------|------|
| 별도 협업 레포 분리 | AI 에이전트 컨텍스트 단절이 치명적 |
| 자체 권한 관리 시스템 구축 | "zero infra" 원칙 위반. Git 내장 기능으로 충분 |
| GitHub Free 플랜에서 완벽한 경로별 보호 약속 | 기술적으로 불가능. 대신 행동 설계로 보완 |

### §2 Step 4: 되돌릴 수 있는가?

| 전환 방향 | 난이도 | 이유 |
|----------|--------|------|
| 통합 → 분리 | **쉬움** | git subtree로 `docs/`를 별도 레포로 추출 가능 |
| 분리 → 통합 | **어려움** | 두 레포의 히스토리 병합이 복잡, 참조 링크 깨짐 |

**Two-way door지만 방향에 따라 난이도가 다르다.** 통합 상태에서 시작하는 것이 유리한 출발점.

### §3 적용 — PRD에 빠진 시나리오

> **STOP**: PRD에 이 시나리오가 빠져 있다

현재 프로토타입의 온보딩: "GitHub 레포 URL 입력 → collaborator 초대 → .claude/ 생성"

빠진 것들:
- 레포가 소스코드 레포일 때의 권한 가이드
- 비개발자의 write 워크플로우 정의
- GitHub 플랜별 보호 수준 차이 안내

**§3 Step 3 — 이번 버전의 최소 범위 (추가해야 할 것):**

1. 온보딩 시 "이 레포에 소스코드가 있는가?" 확인
2. 소스코드 레포 → CODEOWNERS 자동 생성 + branch protection 체크 권장
3. 비개발자 워크플로우 → `/save` 커맨드 + GitHub 웹 UI (docs/에만 쓰기)
4. GitHub 플랜별 추천: Free(branch protection + CODEOWNERS), Team(Push Rulesets 추가)

**§3 Step 4 — 리스크:**

| 가정 | 틀리면 | 검증 방법 |
|------|--------|----------|
| "비개발자는 /save만 쓰니까 src/를 건드릴 일이 없다" | 비개발자가 Claude에게 "이 파일 수정해줘" → Claude가 src/ 건드림 | CLAUDE.md 룰에 "비개발자 세션에서는 src/ 수정 금지" 추가 |
| "CODEOWNERS + branch protection이면 충분하다" | 비개발자가 PR 없이 직접 push하면 보호 안 됨 | branch protection에서 main 직접 push 차단 필수 |
| "소스코드 레포 통합이 모든 팀에 맞다" | 규제 산업 팀은 보안 정책상 불가 | 별도 레포 옵션을 예외 경로로 제공 |

### §7 적용 — 핸드오프와 행동 설계

> **THINK**: 이 문제는 기술 문제가 아니라 행동 설계 문제다

**질문을 바꾸자:**
- ❌ "비개발자가 src/를 건드릴 수 있는가?"
- ✅ **"비개발자가 src/를 건드릴 이유가 있는가?"**

TeamBrain의 워크플로우 설계를 보면:
- PM → `/save` → `docs/roles/pm/traces/`에만 쓰기
- Designer → `/save` → `docs/roles/design/traces/`에만 쓰기
- CLAUDE.md 룰이 역할별 작업 범위를 제한 ("PM 세션에서는 소스코드 수정 금지")
- 비개발자가 소스코드를 건드릴 **동기 자체가 없다**

**권한의 3단계 방어선:**

```
1차: 프로덕트 워크플로우 설계
    /save는 docs/에만 쓰고, CLAUDE.md 룰이 역할별 범위 제한
    → 비개발자가 src/를 건드릴 이유 자체를 없앰

2차: Git branch protection + CODEOWNERS
    main 직접 push 차단, PR 필수, src/ 변경 시 Eng 리뷰 필수
    → 실수로 건드려도 머지 전에 잡힘

3차: Push Rulesets (GitHub Team 플랜)
    경로별 push 차단 — src/ 변경 자체를 서버에서 거부
    → 기술적으로 완벽한 차단 (유료 플랜 필요)
```

**"채널 선택" trace의 인사이트가 여기서도 반복된다**: "도구를 만들자"가 아니라 "행동을 설계하자." 권한 문제의 해결책은 기술적 제한이 아니라, **사람이 올바른 경로로 자연스럽게 유도되는 워크플로우 설계**다. CLAUDE.md 룰 자체가 권한 관리 메커니즘이다.

## 결정사항

| 결정 | 이유 |
|------|------|
| 소스코드 레포 통합을 기본 추천으로 | AI 에이전트 컨텍스트 유지가 핵심 가치. 별도 레포는 이 가치를 훼손 |
| 워크플로우 설계가 1차 권한 방어선 | /save는 docs/에만 쓰고, CLAUDE.md 룰이 역할별 범위 제한. 기술적 제한보다 행동 설계가 먼저 |
| CODEOWNERS + branch protection 가이드 제공 | 2차 안전장치. 비개발자 PR이 src/ 건드리면 Eng 리뷰 필수 |
| PRD에 레포 유형 + 권한 가이드 온보딩 단계 추가 | 현재 빠져 있는 시나리오. 프로덕트 퍼널 ③의 구체적 실행 경로 |
| 별도 레포는 예외 경로로만 | 규제 산업/엄격한 접근 정책 팀용. 기본 추천이 아님 |

## 인사이트

- **"권한 문제는 기술 문제가 아니라 행동 설계 문제"** — GitHub가 경로별 권한을 지원하지 않더라도, 워크플로우가 자연스럽게 사람을 올바른 경로로 유도하면 충분하다. CLAUDE.md 룰 자체가 권한 관리 메커니즘이다. 이건 "채널 선택" trace의 "도구가 아니라 행동 변화" 인사이트와 동일한 패턴이다.
- **AI 에이전트 컨텍스트가 레포 구조의 새로운 결정 변수** — 전통적 레포 분리 논의에서는 "깔끔한 분리 = 좋은 것"이었다. AI 에이전트 시대에는 **레포 분리 = 컨텍스트 단절**이 된다. 이건 레포 구조뿐 아니라 마이크로서비스 vs 모노레포 논쟁 전체에 적용되는 새로운 렌즈다.
- **PRD에 빠진 시나리오 발견** — "어떤 레포에 통합하는가" + "비개발자 권한"은 프로덕트 퍼널 ③의 핵심 실행 단계인데 현재 없음. 이 분석이 PRD 업데이트의 입력이 된다.
- **GitHub 권한 모델의 한계를 프로덕트 설계로 우회** — GitHub가 경로별 write를 지원하지 않는 건 바꿀 수 없다. 하지만 "비개발자가 src/를 건드릴 이유 자체를 없애는 워크플로우"는 우리가 만들 수 있다.

## 다른 팀원이 알아야 할 것

- **Engineering (건희)**: PM 관점에서 CODEOWNERS 자동 생성, branch protection 체크 등이 온보딩에 필요할 수 있다고 봄. 구체적 구현 방식은 엔지니어링 판단에 맡김.
- **전원**: 이 분석은 PRD에 반영해야 할 빠진 시나리오를 발견함. 프로덕트 퍼널 ③("Git 레포에 .claude/ 연결")의 구체적 실행 경로에서 권한/레포 유형 문제가 다뤄져야 함.
