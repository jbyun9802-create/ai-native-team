# 왜 Git + CLAUDE.md 채널로 좁혔는가

## 날짜
2026-04-05

## 목적
- 여러 솔루션 채널 중 "Git + CLAUDE.md"로 좁힌 의사결정 과정을 pm-thinking.md 프레임워크(§2 우선순위, §3 PRD 스코프)에 매핑하여 사고 흐름을 기록한다
- 팀원이 "왜 이 채널인가"의 논리를 따라갈 수 있게 한다

## 핵심 내용 — PM 프레임워크 기반 사고 흐름

### §2 Step 1: 이번에 반드시 달성해야 하는 한 가지 목표는?

> 팀원이 의사결정 시 사고 과정을 기록하게 만드는 것

여기서 핵심 전환이 일어남:
- 처음에는 "사고를 공유하는 도구를 만들자"로 접근
- 그런데 솔루션 방향 자체가 **"사고의 강제"**임을 인식
- 도구가 아니라 **행동 변화**를 만드는 것이 진짜 목표
- 그렇다면 질문이 바뀜: "사고를 강제하는 가장 쉬운 방법은 무엇인가?"

### §2 Step 2: 후보 평가 (RICE)

후보군: 별도 앱 / Notion 플러그인 / Slack 봇 / **Git + CLAUDE.md**

| 기준 | 별도 앱 | Notion/Slack | Git + CLAUDE.md |
|------|---------|-------------|-----------------|
| **Reach** | 높음 (범용) | 중간 (특정 도구 유저) | Claude를 공동 OS로 쓰는 팀 → 모든 작업의 진입점 |
| **Impact** | 높으나 불확실 | 중간 | CLAUDE.md 룰 → AI가 사고를 강제 → 우회 어려움 |
| **Confidence** | 낮음 (만들어봐야 앎) | 중간 | **높음 — CLAUDE.md 메커니즘이 이미 존재함** |
| **Effort** | 높음 (신규 개발) | 중간 (통합 개발) | **최소 — .md 파일 작성으로 즉시 적용** |

**결정적 발견**: "사고를 강제하는 가장 쉬운 방법"을 찾는 과정에서, CLAUDE.md라는 메커니즘이 이미 존재한다는 것을 인식함.
- CLAUDE.md에 룰을 쓰면 Claude가 그 룰을 따름
- 새 인프라를 구축할 필요가 없음
- Confidence와 Effort에서 다른 후보와 압도적 차이

**공동 멘탈 모델 저장소를 Git으로 간 이유**:
- MVP 타겟이 "Claude를 공동 OS로 사용하는 프로덕트 팀" (아래 타겟 선정 참고)
- 이 타겟에게 Git은 이미 일상적 도구 — 새로운 도구 도입 비용 = 0
- Git의 버전 관리가 사고 과정의 이력 추적에 자연스럽게 맞음

### §2 Step 3: "안 하는 것" 명시

| 안 하는 것 | 이유 |
|-----------|------|
| 별도 웹앱/대시보드 구축 | CLAUDE.md로 충분, 새 인프라는 검증 후 |
| Claude를 공동 OS로 쓰는 프로덕트 팀 이외 타겟 | dogfooding 검증이 우선 |
| 범용 협업 도구 | Git + CLAUDE.md 채널에 집중 |

웨비나에서 이 방향을 프로덕트 팀에게 넛지할 예정 — 타겟 확보와 방향 푸시를 동시에.

### §2 Step 4: 되돌릴 수 있는가?

- **Two-way door**: 채널 선택이 틀리면 CLAUDE.md 내용을 바꾸거나 다른 채널로 전환 가능
- .md 파일 기반이라 전환 비용이 거의 없음
- 따라서: **빠르게 결정, 실행하면서 배우기**

### MVP 타겟 선정 — dogfooding (§1 Step 1 재적용)

> "문제가 진짜인가?" 검증의 가장 빠른 방법은 무엇인가?

MVP 타겟을 **"Claude를 공동 OS로 쓰는 프로덕트 팀"**으로 잡은 이유:
- 우리 팀 자체가 이 타겟에 해당함
- 우리가 베타 유저가 되면:
  - 프로덕트를 쓰면서 사고 과정 전달이 실제로 개선되는지 **직접 체감**
  - 문제 정의가 틀렸으면 **바로** 알 수 있음
  - 사용자 피드백 루프가 0에 수렴
- §1의 "데이터가 있는가, 가설인가?" 질문에 대한 답: **dogfooding으로 가설→데이터 전환**
- 타겟 선정과 검증 전략이 하나의 결정으로 연결됨: dogfooding = 타겟이자 검증

### 스코프 확인 (§3)

**§3 Step 1 — 프레스 릴리즈**:
- "팀의 사고 과정이 AI를 통해 자동으로 기록되고 공유되는 세상"
- 한 문장: CLAUDE.md에 사고 강제 룰을 넣으면, 팀원이 Claude와 일하는 모든 순간에 사고가 구조화됨

**§3 Step 3 — 이번 버전의 최소 범위**:
- CLAUDE.md에 STOP/THINK 룰 정의
- Git 레포에 traces 구조 세팅
- 팀 내부에서 2주 사용 후 검증
- "이것만 있으면 가치를 전달할 수 있다": CLAUDE.md 룰 + traces 폴더 + 팀원 3명의 실사용

**§3 Step 4 — 리스크**:
- 가정: "CLAUDE.md 룰이 실제로 사고를 강제한다"
- 틀리면: 사고 기록이 형식적이 됨 (체크리스트 채우기식)
- 검증 방법: 2주 dogfooding 후 traces 품질 리뷰

## 결정사항

| 결정 | 이유 |
|------|------|
| 솔루션 채널: Git + CLAUDE.md | 사고 강제 메커니즘(CLAUDE.md)이 이미 존재, 새 인프라 불필요 |
| MVP 타겟: Claude를 공동 OS로 쓰는 프로덕트 팀 | dogfooding으로 가설 검증 가능, 우리가 베타 유저 |
| 공동 멘탈 모델 저장소: Git | 타겟에게 이미 일상 도구, 버전 관리가 사고 이력 추적에 적합 |

## 인사이트

- **"가장 쉬운 방법" 질문이 답을 줬다**: 솔루션의 핵심이 "새 도구 구축"이 아니라 "기존 메커니즘(CLAUDE.md)을 활용한 행동 변화 설계"임을 명확히 함. RICE에서 Confidence와 Effort가 압도적으로 유리한 선택지가 하나뿐이었음.
- **타겟 = 검증**: 타겟 선정과 검증 전략이 분리된 결정이 아니라 하나로 연결됨. "Claude를 공동 OS로 쓰는 팀" = "우리가 직접 검증할 수 있는 팀".
- **솔루션 방향의 전환**: "도구를 만들자" → "사고를 강제하자" → "이미 있는 메커니즘으로 강제하자"의 사고 흐름이 이 결정의 핵심.

## 다른 팀원이 알아야 할 것

- **전원**: 채널 선택이 "우리가 Git을 좋아해서"가 아니라 PM 프레임워크의 RICE → 스코프 → Two-way door 과정을 밟은 결과임. 이 trace를 읽으면 논리 흐름을 따라갈 수 있음.
