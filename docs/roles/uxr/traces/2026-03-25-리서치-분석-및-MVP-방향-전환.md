# UXR 리서치 분석 및 MVP 방향 전환

## 날짜
2026-03-25

## 목적
인터뷰 4건 정리 → 교차 분석 → MVP/웨비나 방향 권고사항 도출

## 내가 내린 판단과 그 이유

### 판단 1: 노트 먼저, 분석은 그 다음
분석부터 할 수도 있었지만 개별 노트를 먼저 구조화하는 방향을 선택. 이유: 교차 분석은 개별 인터뷰 해석에 의존하므로, top-down으로 가면 데이터를 선택적으로 읽게 될 위험이 있음.

### 판단 2: 자가 인터뷰 추가
원래 플랜에는 3명(PM, UXD, Eng)만 포함되어 있었으나, 리서처 본인도 이 팀의 일원이고 같은 핸드오프 문제를 겪고 있으므로 자가 인터뷰를 추가. 편향 리스크는 인지하고 있으며 문서에 명시.

### 판단 3: PM 트랜스크립트 복원
처음 받은 PM 트랜스크립트에 ~120줄 갭이 있었음. 이후 전체 트랜스크립트를 다시 받아 교체. 이유: PM 인터뷰가 가장 풍부한 데이터 소스였고, 갭 구간에 traces 시스템 경험, 병렬 작업 충돌 등 핵심 에피소드가 포함되어 있었음. 불완전한 데이터로 분석하면 PM 관점이 과소 대표됨.

### 판단 4: MVP가 Git+Claude Code로 수렴하는 것에 대한 의문 제기
교차 분석 후 MVP 스코프를 정리하는 과정에서, 팀이 "당연히 Git + MD 파일 기반"으로 가고 있다는 것을 인지. **이것은 문제 검증에서 나온 결론이 아니라 도구 친숙성에서 나온 가정.**

리서치가 검증한 것은 "reasoning이 전달되지 않는다"는 **문제**이지, "Git에 MD 파일로 저장하면 해결된다"는 **해결책**이 아님.

이 구분이 중요한 이유:
- PM의 traces가 작동한 것은 PM 1명 + 인턴(상하관계)에서만 검증됨
- UXR 본인도 reasoning의 90%를 머릿속에 둔다고 인정 — "가벼운 트리거"가 실제로 가벼운지 미검증
- UXD는 "설명이 길어질수록 리스크가 올라간다"고 언급 — reasoning 문서가 또 다른 해석 대상이 될 수 있음
- 비개발자에게 Git은 접근성 장벽

**이에 따라 MVP 권고에 "도구 만들기 전에 2주 수동 실험을 먼저 수행"을 포함.** 핸드오프 시 3문장만 남기는 실험으로 행동 발생 여부를 먼저 확인할 필요가 있음.

### 판단 5: 웨비나와 MVP 권고 분리
처음에는 교차 분석 안에 웨비나 제안이 MVP와 혼재되어 있었으나 이를 분리.
- MVP는 우리 팀의 문제를 푸는 것 — 성공 지표: 재작업 감소 여부
- 웨비나는 다른 팀의 공감과 행동 유도 — 성공 지표: 공감도, 커뮤니티 유입
- 같은 메시지처럼 보이지만 청중이 다르고 목적이 다름
- 웨비나가 Claude Code 튜토리얼이 되면 안 됨 — 메시지는 "도구가 아니라 설계"

## 이 세션에서 알게 된 것

**리서치가 증명한 문제를 팀 자신이 그대로 재현하고 있었음.**
- 이 trace의 첫 번째 버전은 "무엇을 했는지"만 기록하고 "왜 그렇게 했는지"가 빠져 있었음
- MVP 방향을 전환시킨 reasoning("도구 편향을 경계해야 한다")이 문서에 캡처되지 않았음
- 이것 자체가 "reasoning이 산출물과 함께 저장되지 않는다"는 핵심 발견의 살아있는 증거

**MVP 실험에 대한 시사점:** reasoning 캡처에는 의식적 노력이 필요. "자동 추출"이라는 MVP 가정이 유효한지 검증이 필요. 이 trace도 지적하지 않았으면 reasoning 없는 changelog로 남았을 것.
