# UXR 리서치 분석 및 MVP 방향 전환

## 날짜
2026-03-25

## 목적
인터뷰 4건 정리 → 교차 분석 → MVP/웨비나 방향 권고사항 도출

## 내가 내린 판단과 그 이유

### 판단 1: 노트 먼저, 분석은 그 다음
분석부터 할 수도 있었는데 개별 노트 먼저 정리함. top-down으로 가면 보고 싶은 것만 보게 됨. 교차 분석은 개별 해석이 탄탄해야 의미 있음.

### 판단 2: 자가 인터뷰 추가
원래 플랜엔 3명(PM, UXD, Eng)만 있었음. 근데 나도 이 팀 멤버이고 같은 핸드오프 문제 겪고 있는데 내 관점 빠지면 불완전함. 편향 리스크는 알고 있고 문서에 명시해둠.

### 판단 3: PM 트랜스크립트 복원
처음 받은 PM 트랜스크립트에 ~120줄 갭 있었음. 나중에 전체본 다시 받아서 교체함. PM 인터뷰가 가장 풍부한 데이터 소스였고, 빠진 구간에 traces 경험, 병렬 작업 충돌 같은 핵심 에피소드가 있었음. 불완전한 데이터로 분석하면 PM 관점이 과소 대표됨.

### 판단 4: MVP가 Git+CC로 수렴하는 거 의문 제기
교차 분석 끝나고 MVP 스코프 정리하는데, 팀이 "당연히 Git + MD 파일"로 가고 있었음. **이건 문제 검증에서 나온 결론이 아니라 도구 친숙성에서 나온 가정임.**

리서치가 검증한 건 "reasoning이 전달 안 된다"는 **문제**지, "Git에 MD로 저장하면 해결된다"는 **해결책**이 아님.

왜 이게 중요하냐면:
- PM의 traces가 작동한 건 PM 1명 + 인턴(상하관계)에서만 검증됨
- 나 자신도 reasoning의 90%를 머릿속에 둔다고 인정함 — "가벼운 트리거"가 진짜 가벼운지 모름
- UXD는 "설명이 길어질수록 리스크 올라간다"고 함 — reasoning 문서가 또 다른 해석 대상이 될 수 있음
- 비개발자한테 Git은 접근성 장벽임

**그래서 MVP 권고에 "도구 만들기 전에 2주 수동 실험부터"를 넣음.** 핸드오프할 때 3문장만 남겨보는 실험으로 행동이 일어나는지부터 확인해야 함.

### 판단 5: 웨비나와 MVP 권고 분리
처음엔 교차 분석 안에 웨비나 제안이랑 MVP가 섞여 있었음. 분리한 이유:
- MVP는 우리 팀 문제 해결 — 성공 지표: 재작업 줄었는가
- 웨비나는 다른 팀의 공감과 행동 유도 — 성공 지표: 공감도, 커뮤니티 유입
- 같은 메시지처럼 보이는데 청중이 다르고 목적이 다름
- 웨비나가 CC 튜토리얼이 되면 안 됨 — 메시지는 "도구가 아니라 설계"

## 이 세션에서 알게 된 것

**리서치가 증명한 문제를 우리가 그대로 재현하고 있었음.**
- 이 trace 첫 버전은 "뭘 했는지"만 기록하고 "왜 그렇게 했는지"가 빠져 있었음
- 내가 MVP 방향 전환시킨 reasoning("도구 편향 경계해야 함")이 문서에 캡처 안 됐었음
- 이거 자체가 "reasoning이 산출물과 함께 저장 안 된다"는 핵심 발견의 살아있는 증거임

**MVP 실험에 시사하는 것:** reasoning 캡처는 의식적 노력이 필요함. "자동 추출"이라는 MVP 가정이 맞는지 검증 필요함. 이 trace도 내가 지적 안 했으면 reasoning 없는 changelog로 남았을 것.
