# Study 001 — AI-네이티브 팀 협업 워크플로우 리서치

> **상태:** 🟢 분석 완료 | **리서처:** Ahrom Kim
> **이해관계자:** Robin (PM/GTM), 우석 (Design), 건희 (Engineering)
> **최종 수정일:** 2026-03-25

---

## 1. 배경

우리 팀은 AI-네이티브 웨비나를 준비하면서, 직접 겪고 있는 문제를 프로덕트로 해결하려 하고 있다. 핵심 문제는 **"개인이 AI를 쓰는 방법은 알지만, AI가 만든 것을 다음 사람에게 어떻게 넘길지 모른다"**는 것이다. 팀원들이 각자 AI와 대화하는데 그 컨텍스트가 개인 채팅창에서 사라지고, 상대방의 고민, 우선순위, 의사결정 근거가 전달되지 않는다.

문제 분석(problem-analysis.md) 결과, 7개의 하위 문제가 식별되었고 하나의 근본 원인으로 수렴한다: **의사결정 맥락(context)이 결과물과 함께 저장되지 않는다.** MVP는 "근거가 붙은 핸드오프"를 핵심 컨셉으로 설정했다.

현재까지 팀 미팅 4회를 진행했으며, 그중 2회는 각 팀원이 자신의 AI 워크플로우를 직접 시연하는 세션이었다. 기본적인 워크플로우 파악은 완료된 상태이므로, 이 리서치는 이미 공유된 내용을 기반으로 **핸드오프 시 컨텍스트 유실 경험**과 **AI-네이티브 팀에 대한 멘탈 모델**을 심층적으로 파악하는 데 집중한다. 동시에 웨비나 콘텐츠 개발과 프로덕트 MVP 방향 검증을 지원한다.

## 2. 목표

1. 이미 공유된 각 역할의 AI 워크플로우를 바탕으로, 워크플로우 간 접점과 단절 지점을 구체화
2. 역할 간 핸드오프 시 컨텍스트가 유실되는 구체적 지점과 패턴을 식별
3. "AI-네이티브 팀"에 대한 각 역할의 멘탈 모델과 기대 수준을 이해
4. 웨비나 내러티브와 프로덕트 MVP 방향을 검증할 수 있는 인사이트를 확보

## 3. 리서치 질문

1. 워크플로우 시연에서 드러난 각 역할의 AI 활용 방식 중, 다른 역할과의 접점에서 어떤 마찰이 발생하는가?
2. AI로 생성한 산출물이 다음 역할로 전달될 때, 어떤 맥락(reasoning)이 유실되는가?
3. 현재 팀의 공유 레포(Git 기반 지식 저장소)가 컨텍스트 공유에 얼마나 효과적인가? 한계는?
4. 각 역할이 생각하는 "AI 도구 사용"과 "AI-네이티브 팀"의 차이는 무엇인가?
5. 역할 간 효과적 협업을 위해 상대방에게 필요하다고 생각하는 AI 관련 지식 수준은?
6. 핸드오프 시 "근거가 붙은 산출물"이 있었다면 어떤 상황이 달라졌을 것인가?
7. 웨비나 청중에게 가장 공감을 얻을 수 있는 데모 또는 사례는 무엇인가?

## 4. 방법론

이 연구는 **이해관계자 인터뷰 + 워크플로우 매핑**의 혼합 방법론을 사용한다.

| 항목 | 내용 |
|---|---|
| **방법** | 반구조화 인터뷰 + 워크플로우 매핑 |
| **소요 시간** | 인터뷰 1인당 45-60분 |
| **장소/플랫폼** | 원격 (Zoom 또는 Teams) |
| **데이터 소스** | 인터뷰 녹취록, 기존 미팅 4회 기록 (워크플로우 시연 2회 포함), GitHub 레포 내 기존 문서(traces/, meetings/) |

### 방법론 상세

**1단계: 사전 분석 (인터뷰 전)** — 부분 완료
- 기존 미팅 4회 기록 분석 (킥오프, 워크플로우 시연 2회, 문제 선정)
- 워크플로우 시연 세션에서 각 역할의 AI 활용 패턴 및 도구 정리
- GitHub 레포의 `docs/roles/` 내 각 역할 README 및 traces 폴더 리뷰
- `docs/shared/meetings/` 내 미팅 기록 교차 확인

**2단계: 이해관계자 인터뷰**
- 각 역할별 맞춤형 인터뷰 가이드 사용 (7번 섹션 참조)
- 인터뷰 중 실제 AI 사용 화면 또는 산출물 공유 요청 (직접 시연)
- 핸드오프 실패/성공 사례의 구체적 에피소드 수집

**3단계: 역할 간 교차 분석**
- 역할 간 핸드오프 흐름도 작성 (PM→디자인, PM→개발, 디자인→개발)
- 컨텍스트 유실 지점 매핑
- 공통 페인포인트 vs. 역할별 페인포인트 분류

4. 제품 데모 요소가 있나요, 아니면 순수하게 프로세스/워크플로우 쇼케이스인가요?
5. 이 프로젝트에서 "완료"는 어떤 모습인가요? 웨비나 외에 추가로 만들어야 할 것은?

**킥오프: 타임라인 & 마일스톤**

**킥오프: 역할 & 의사결정 권한**
8. 최종 내러티브와 스크립트의 오너는 누구인가요? 승인 권한은?
9. 발표를 어떻게 나눌 건가요? — 각자 자기 역할 관점에서 발표? 한 명이 내레이션?

**킥오프: 청중 & 메시징**
11. 청중이 정확히 누구인가요? (회사 규모, 직급, AI 성숙도)
12. 청중이 일주일 후에도 기억했으면 하는 한 문장은?

16. 워크플로우 시연 이후 바뀌거나 추가된 AI 활용 방식이 있나요?
17. AI로 생성한 결과물(PRD, 유저 스토리, 경쟁 분석 등) 중에서 팀에게 전달하기 가장 어려웠던 건 뭐였나요?

**핸드오프 & 협업 페인포인트**
18. 스펙이나 의사결정을 디자인/개발에 넘길 때, 어떤 컨텍스트가 주로 유실되나요?
19. AI로 생성한 산출물이 다른 팀원에게 잘못 해석된 적이 있나요? 무슨 일이 있었나요?
20. 의사결정의 *근거*를 어떻게 공유하고 있나요? — 결과물만이 아니라 이유까지요.

**"AI-네이티브"란 무엇인가**
21. AI 도구를 *사용하는* 팀과 *AI-네이티브* 팀의 차이가 뭐라고 생각하세요?
22. 현재 워크플로우에서 AI가 해결해야 하는데 아직 못 하고 있는 가장 큰 병목은?

**지식 & 역량**
23. AI-네이티브 팀에서 PM이 갖춰야 할 AI 지식 수준은? (프롬프트 엔지니어링? 모델 한계 이해? API 기초?)
24. 디자이너나 엔지니어가 당신의 AI 활용 방식에 대해 더 잘 이해해줬으면 하는 점은?

11. 그리고 이 과정에서 AI가 어떻게 도움을 줄 수 있다고 생각하실까요?
12. 여기에서 AI로 인해 유실되는 부분은 무엇이라고 생각하시나요?

# 연구 후 (현장 조사 완료 후 작성)

## 9. 분석 & 권고사항

### 요약

- **핵심 발견:** 4명 전원이 "산출물은 전달되지만 reasoning은 전달되지 않는다"에 수렴. 이는 AI 도입으로 악화됨 — AI 산출물은 인간이 만든 것보다 근거가 더 불투명함.
- **AI-네이티브의 정의:** "도구 사용 vs 시스템 설계"로 4/4 합의. PM은 "에이전틱 워크플로우가 협업에 내장된 팀", UXD는 "80/20 위임을 위한 메타 인지", UXR은 "AI를 팀원으로 대화하는가"로 표현.
- **공통 전제:** 본업 역량이 AI 활용의 선행 조건이라는 점에 전원(4/4) 동의. AI는 곱하기 — 병목이 있으면 병목도 곱해짐.

### 권고사항

1. **MVP는 "근거가 붙은 핸드오프"에 집중해야 한다.** 현재 traces/save 시스템이 PM에게만 작동 중이며, 이를 전 역할로 확장하는 것이 가장 높은 ROI를 가진 제품 방향이다.
2. **역할 경계 협약은 도구가 아닌 대화로 선행되어야 한다.** 접근 권한(read/write), 컨벤션(MD 파일 제안은 되지만 코드는 안 됨)을 팀이 함께 합의하는 세션이 제품 개발 전에 필요하다.
3. **Eng 보완 인터뷰를 최우선으로 진행해야 한다.** 트랜스크립트 손상으로 개발자 관점 데이터가 불충분하며, 이는 MVP 기술 설계와 웨비나 Eng 파트 모두에 영향을 미친다.
