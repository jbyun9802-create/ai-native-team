# TeamBrain Assessment — 질문 구성 + 출력 구조

## 개요

| 항목 | 내용 |
|------|------|
| 총 문항수 | ~88문항 (프로파일 3 + 리터러시 ~25 + 강제선택 60) |
| 소요 시간 | ~20분 |
| 측정 축 | 6축 (AI 협업 4축 + 팀 협업 2축) + AI 리터러시 레벨 |
| 테스트 구조 | 4 Phase: 프로파일 → 리터러시 → 강제선택 60쌍 → 확인 |
| 출력 | 개인 리포트 + {이름}.md → zip 다운로드. 팀: team-report.md + CLAUDE.md + 전원 {이름}.md → 오너에게 이메일 |
| 비용 | 무료 (API 비용 우리 부담, ~120원/건) |
| 인프라 | Vercel + Supabase(임시저장+매직링크) + Resend(이메일) + Claude API |

---

## Part 1: 질문 구성

### Section A: 프로파일 (3문항)

```
A1. 이름: [텍스트 입력]

A2. 역할:
    ○ PM  ○ Engineering  ○ Design  ○ UX Research  ○ Growth (Sales/Marketing)

A3. Seniority:
    ○ Junior (1-3년)  ○ Mid (3-7년)  ○ Senior (7년+)  ○ Lead
```

→ 역할 선택에 따라 Section B(리터러시)와 리포트가 직무별 분기.

### Section B: AI 리터러시 — 직무별 경험 체크 (~25문항, 체크박스)

"해본 적이 있다" 에 체크. 체크 수 + 단계별 위치로 리터러시 레벨 산출.

#### PM

**기초 활용:**
- □ AI에게 미팅 녹음/노트를 주고 요약을 시켜본 적이 있다
- □ AI에게 이메일/슬랙 메시지 초안을 작성시켜본 적이 있다
- □ AI에게 자료를 주고 핵심 요약을 뽑아본 적이 있다
- □ AI에게 브레인스토밍 아이디어를 뽑아달라고 해본 적이 있다

**업무 적용:**
- □ AI에게 유저 문제를 설명하고 PRD 구조를 잡아본 적이 있다
- □ AI가 작성한 문서에서 빠진 엣지케이스를 발견하고 추가해본 적이 있다
- □ AI에게 경쟁사 자료를 분석시켜 포지셔닝 맵을 만들어본 적이 있다
- □ AI에게 유저 인터뷰 트랜스크립트를 주고 패턴을 추출시켜본 적이 있다
- □ AI에게 같은 기능의 유저 스토리를 여러 관점으로 작성시켜 비교해본 적이 있다
- □ AI에게 내 PRD의 논리적 허점을 지적해달라고 해본 적이 있다

**고급 활용:**
- □ AI에게 역할(페르소나)을 부여하고 특정 관점에서 피드백을 받아본 적이 있다
- □ AI에게 few-shot 예시를 주어 원하는 형식/톤의 결과를 얻어본 적이 있다
- □ 하나의 작업을 여러 프롬프트 전략으로 시도해서 결과를 비교해본 적이 있다
- □ AI의 결과가 틀렸을 때 왜 틀렸는지 분석하고 프롬프트를 개선해본 적이 있다
- □ AI에게 가상 유저 페르소나를 만들어서 제품/기능을 테스트해본 적이 있다

**Eval (AI 평가):**
- □ AI 모델 여러 개의 결과물을 같은 업무로 비교 평가해본 적이 있다
- □ AI 결과물의 품질을 체계적으로 측정하는 기준을 정의해본 적이 있다
- □ AI 도구의 ROI(시간 절약, 품질 변화)를 측정해본 적이 있다

**프로토타입 개발:**
- □ Claude Code/Cursor 등으로 동작하는 프로토타입을 직접 만들어본 적이 있다
- □ AI를 이용해 데이터 분석 스크립트(Python, SQL 등)를 작성하고 실행해본 적이 있다
- □ AI로 웹페이지/랜딩페이지를 직접 만들어본 적이 있다

**워킹 프로덕트 배포:**
- □ AI를 이용해 만든 도구/서비스를 실제로 배포한 적이 있다
- □ AI로 만든 자동화를 팀이 실제로 사용하고 있다

**시스템 설계:**
- □ CLAUDE.md나 시스템 프롬프트를 직접 작성하여 AI 행동을 커스터마이즈해본 적이 있다
- □ AI를 활용한 반복 업무 워크플로우를 설계하고 팀에 공유해본 적이 있다

**끝판왕:**
- □ AI agent가 자동으로 데이터 수집 → 분석 → 리포트 생성하는 파이프라인을 구축해본 적이 있다
- □ 여러 AI agent를 연결해서 end-to-end 업무 자동화를 만든 적이 있다
- □ MCP 서버나 커스텀 AI 도구를 직접 개발해서 팀에 배포한 적이 있다
- □ 슬래시 커맨드, hooks 등을 직접 설계하여 팀의 AI 개발 환경을 세팅한 적이 있다

#### Engineering

#### 리터러시 레벨 산출

```
체크 분포 → 레벨:
  기초만 체크        → Level 1 (입문)
  업무 적용까지      → Level 2 (활용)
  고급 활용까지      → Level 3 (숙련)
  Eval/시스템 설계   → Level 4 (전문가)
  끝판왕 1개 이상    → Level 5 (아키텍트)
```

### Section C: 강제 선택 — AI 협업 4축 (40쌍)

타이머 없음. 5점 척도 (강하게 왼쪽 / 왼쪽 / 중립 / 오른쪽 / 강하게 오른쪽).
축 순서는 섞어서 출제.

#### 축 1: 판단 기점 — Self-Directed (S) vs AI-Directed (A) · 10쌍

| # | 왼쪽 → S | 오른쪽 → A |
|---|----------|-----------|
| 1 | AI에게 물어보기 전에 내 생각을 정리하는 편이다 | AI에게 먼저 물어보고 거기서 실마리를 잡는 편이다 |
| 2 | 초안은 내가 쓰고, AI에게 보완점을 묻는다 | AI가 초안을 만들면, 내가 방향을 잡아준다 |
| 3 | 내 가설이 있어야 AI에게 질문할 수 있다 | AI의 답변을 보면서 내 가설이 만들어진다 |
| 4 | 방향이 정해진 다음에 AI를 쓰는 게 효율적이다 | 방향을 정하는 과정에서 AI를 쓰는 게 효율적이다 |
| 5 | AI가 예상 밖의 답을 주면 내 기준으로 다시 판단한다 | AI가 예상 밖의 답을 주면 그 관점을 탐색해본다 |
| 6 | 프로젝트 시작 시 나만의 계획을 먼저 세운다 | 프로젝트 시작 시 AI에게 여러 접근법을 뽑아달라고 한다 |
| 7 | AI에게 맡기더라도 판단 기준은 내가 미리 정한다 | AI에게 맡기면서 판단 기준도 함께 만들어간다 |
| 8 | 내 전문 분야에서는 AI 없이도 첫 방향을 잡을 수 있다 | 내 전문 분야에서도 AI의 시각이 출발점을 넓혀준다 |
| 9 | AI를 쓸 때 '내가 원하는 것'이 명확한 상태에서 시작한다 | AI를 쓸 때 '뭐가 가능한지'를 먼저 파악하면서 시작한다 |
| 10 | 결론이 AI 제안과 다르더라도 내 판단을 신뢰한다 | 내 판단과 다르더라도 AI 제안의 근거를 먼저 살핀다 |

#### 축 2: 협업 깊이 — Directive (D) vs Dialogic (G) · 10쌍

| # | 왼쪽 → D | 오른쪽 → G |
|---|----------|-----------|
| 1 | AI에게 한 번에 명확한 지시를 주는 게 낫다 | AI와 여러 번 주고받으며 결과를 발전시키는 게 낫다 |
| 2 | AI에게 맡길 때 입력과 출력 형식을 정해서 준다 | AI에게 맡길 때 맥락을 공유하고 같이 형식을 잡아간다 |
| 3 | 좋은 프롬프트 하나면 원하는 결과를 얻을 수 있다 | 좋은 결과는 여러 턴의 대화에서 나온다 |
| 4 | AI 작업을 요청할 때 기대 결과물을 구체적으로 적는다 | AI 작업을 요청할 때 문제 상황과 고민을 함께 나눈다 |
| 5 | AI가 한 번에 좋은 결과를 주면 가장 만족스럽다 | AI와 대화하면서 처음 생각 못한 결과가 나오면 가장 만족스럽다 |
| 6 | AI에게 일을 시킬 때 체크리스트처럼 나눠서 준다 | AI에게 일을 시킬 때 전체 그림을 공유하고 같이 쪼갠다 |
| 7 | AI와의 작업 시간을 줄이는 것이 생산성이다 | AI와의 작업 깊이를 높이는 것이 생산성이다 |
| 8 | AI 결과가 기대와 다르면 프롬프트를 수정해서 다시 보낸다 | AI 결과가 기대와 다르면 왜 그렇게 나왔는지 대화를 이어간다 |
| 9 | AI에게 역할을 정해주고 그 안에서 일하게 한다 | AI에게 나의 사고 파트너 역할을 기대한다 |
| 10 | 같은 작업이면 AI와의 대화 턴 수가 적을수록 좋다 | 대화 턴이 늘더라도 더 나은 결과가 나오면 괜찮다 |

#### 축 3: AI 이해도 — Strategic (T) vs Experimental (E) · 10쌍

| # | 왼쪽 → T | 오른쪽 → E |
|---|----------|-----------|
| 1 | AI가 잘하는 일과 못하는 일을 구분해서 맡긴다 | AI가 할 수 있을지 모르는 일도 일단 시켜본다 |
| 2 | 검증된 프롬프트 패턴을 반복해서 쓴다 | 새로운 프롬프트 방식을 자주 실험한다 |
| 3 | AI를 쓸 때 예측 가능한 결과가 나오는 게 중요하다 | AI를 쓸 때 예상 못한 결과가 나오는 것도 가치 있다 |
| 4 | 이 상황에 어떤 AI 기능이 맞는지 아는 편이다 | 이 상황에 AI를 어떻게 쓸지 여러 가지 시도해보는 편이다 |
| 5 | AI로 실패하면 그 방법은 피하게 된다 | AI로 실패하면 조건을 바꿔서 다시 시도한다 |
| 6 | AI 활용법을 팀에 공유할 때 '이렇게 하면 확실하다'고 말한다 | AI 활용법을 팀에 공유할 때 '이렇게 해봤는데 재밌다'고 말한다 |
| 7 | AI를 도입할 때 성공 확률이 높은 업무부터 적용한다 | AI를 도입할 때 가장 궁금한 업무에 먼저 적용해본다 |
| 8 | 내 AI 활용 방식은 꽤 안정적인 패턴이 있다 | 내 AI 활용 방식은 계속 바뀌고 있다 |
| 9 | AI의 한계를 알면 그 안에서 최대한 활용한다 | AI의 한계를 만나면 우회 방법을 찾아본다 |
| 10 | AI 도구가 업데이트되면 기존 작업에 영향이 없는지 확인한다 | AI 도구가 업데이트되면 새 기능부터 써본다 |

#### 축 4: 품질 관여 — Craftsperson (C) vs Leverager (L) · 10쌍

| # | 왼쪽 → C | 오른쪽 → L |
|---|----------|-----------|
| 1 | AI가 만든 결과물을 내 스타일로 다시 다듬는다 | AI가 만든 결과물이 괜찮으면 바로 쓴다 |
| 2 | AI 결과물에 내 이름을 걸려면 직접 검토해야 한다 | AI 결과물의 핵심만 맞으면 빠르게 전달하는 게 낫다 |
| 3 | 90점짜리를 95점으로 올리는 데 시간을 쓸 수 있다 | 90점이면 충분하고, 남은 시간에 다른 일을 한다 |
| 4 | AI 결과물을 읽으면서 '여기는 고치면 더 낫겠다'가 바로 보인다 | AI 결과물을 읽으면서 '이 정도면 목적에 맞겠다'를 먼저 판단한다 |
| 5 | AI가 만든 코드나 문서에 내 기준이 반영되어야 한다 | AI가 만든 코드나 문서가 동작하면 스타일은 크게 신경 안 쓴다 |
| 6 | 결과물의 완성도가 내 전문성을 보여준다고 생각한다 | 결과물의 속도와 양이 내 전문성을 보여준다고 생각한다 |
| 7 | AI 결과물을 그대로 보내는 건 불편하다 | AI 결과물에 손을 대다 보면 시간이 아깝다는 생각이 든다 |
| 8 | 동료가 내 결과물의 세부 표현까지 좋다고 하면 뿌듯하다 | 동료가 내 결과물이 빠르게 나와서 좋다고 하면 뿌듯하다 |
| 9 | AI 결과를 받은 뒤 내가 다듬는 시간이 전체 작업의 핵심이다 | AI 결과를 받은 뒤 바로 다음 작업으로 넘어가는 게 핵심이다 |
| 10 | 품질 기준에 못 미치면 AI에게 다시 시킨다 | 일정 기준만 넘으면 다음 단계로 넘어간다 |

#### 축 5: 소통 스타일 — Explicit (E) vs Implicit (I) · 10쌍

AI를 쓰는 동료와의 소통 맥락.

| # | 왼쪽 → E | 오른쪽 → I |
|---|----------|-----------|
| 1 | 팀원에게 작업을 넘길 때 판단 근거와 기각한 대안까지 적어서 보내는 편이다 | 팀원에게 작업을 넘길 때 결론과 다음 할 일 위주로 간결하게 보내는 편이다 |
| 2 | 동료가 AI로 만든 결과물을 보내올 때, 어떤 과정을 거쳤는지 같이 보고 싶다 | 동료가 AI로 만든 결과물을 보내올 때, 결과물 자체가 좋으면 과정은 안 봐도 된다 |
| 3 | 핸드오프 문서에 '확실하지 않은 부분'과 '고민한 흔적'을 남기는 편이다 | 핸드오프 문서에 '확정된 방향'과 '다음 사람이 할 일'을 남기는 편이다 |
| 4 | 팀원이 맥락 없이 결론만 보내면 '왜 이렇게 결정했어?'라고 물어보게 된다 | 팀원이 결론만 보내도 일단 진행하고, 궁금하면 나중에 물어보는 편이다 |
| 5 | AI가 만든 결과물을 팀에 공유할 때, AI를 어떻게 활용했는지도 설명하는 편이다 | AI가 만든 결과물을 팀에 공유할 때, 결과물만 공유하고 방법은 물어보면 알려주는 편이다 |
| 6 | 팀 슬랙에 올리는 업데이트가 길어지더라도 배경 설명을 넣는 게 낫다 | 팀 슬랙 업데이트는 짧을수록 좋고, 깊은 얘기는 미팅에서 하면 된다 |
| 7 | 내가 넘긴 작업에 대해 팀원이 질문할 일이 적도록 문서를 꼼꼼히 쓰는 편이다 | 문서는 빠르게 쓰고, 질문이 오면 그때 구두로 보충하는 게 더 효율적이다 |
| 8 | AI 결과물에 대한 내 판단이 바뀌었을 때, 바뀐 이유를 팀에 공유하는 편이다 | AI 결과물에 대한 내 판단이 바뀌었을 때, 바뀐 결론만 공유하고 넘어가는 편이다 |
| 9 | 동료의 AI 활용 방식이 나와 다르면, 서로의 접근법을 정리해보고 싶다 | 동료의 AI 활용 방식이 나와 다르면, 결과만 좋으면 각자 방식대로 하면 된다 |
| 10 | 작업 중 불확실한 부분이 있으면 그 불확실성 자체를 팀에 미리 공유하는 편이다 | 작업 중 불확실한 부분이 있으면 내가 해결한 뒤에 결과를 공유하는 편이다 |

#### 축 6: 갈등 대처 — Firm (F) vs Bridging (B) · 10쌍

팀 내 의견 충돌·갈등 상황에서의 대처 스타일. AI 무관, 순수 협업 성향 측정.

| # | 왼쪽 → F | 오른쪽 → B |
|---|----------|-----------|
| 1 | 팀원의 작업물에 문제가 보이면 구체적으로 지적하고 대안을 제시하는 편이다 | 팀원의 작업물에 문제가 보이면 상대 의도를 먼저 듣고 같이 조정해가는 편이다 |
| 2 | 팀원과 방향이 다르면, 내 근거를 밝히고 설득하는 편이다 | 팀원과 방향이 다르면, 양쪽 장점을 섞어 절충안을 찾는 편이다 |
| 3 | 회의에서 결과물 품질이 부족하다고 느끼면 바로 말하는 편이다 | 회의에서 결과물 품질이 부족하다고 느끼면 상대 기준을 먼저 물어보는 편이다 |
| 4 | 내가 검토한 방향이 맞다고 확신하면, 반대 의견이 있어도 입장을 유지하는 편이다 | 내가 확신하더라도 반대 의견이 있으면 합의점을 찾으려고 조정하는 편이다 |
| 5 | 동료의 산출물이 부족하면 '이 부분은 고쳐야 한다'고 직접 말하는 편이다 | 동료의 산출물이 부족해도 '이 부분은 어떻게 생각해?'로 풀어가는 편이다 |
| 6 | 팀 기준을 정할 때, 내 경험에 근거한 의견을 강하게 내는 편이다 | 팀 기준을 정할 때, 각자 경험을 들으며 공통 기준을 만들어가는 편이다 |
| 7 | 동료가 리스크를 과소평가한다고 느끼면, 구체적 위험을 짚어 경고하는 편이다 | 동료가 리스크를 과소평가한다고 느끼면, 같이 검증해보자고 제안하는 편이다 |
| 8 | 넘겨받은 작업의 방향이 잘못됐다고 판단하면, 근거와 함께 내 판단을 전달한다 | 넘겨받은 작업의 방향이 잘못됐다고 판단하면, 넘겨준 사람과 대화하며 재조정한다 |
| 9 | 팀 규칙에 동의하지 않으면, 대안을 제시하며 변경을 주장하는 편이다 | 팀 규칙에 동의하지 않으면, 일단 따르면서 실제 사례로 개선을 제안하는 편이다 |
| 10 | 팀원과 품질 기준이 다르면, 내 기준이 왜 맞는지 설명하는 편이다 | 팀원과 품질 기준이 다르면, 서로의 기준을 이해하고 팀 기준을 새로 합의한다 |

### 개인 테스트 완료 시 → zip 다운로드

```
{이름}.zip
  ├── {이름}-report.md    ← 개인 리포트 풀버전 (수 페이지)
  └── {이름}.md           ← 사고 강제 룰 — STOP/THINK 포함
                            → .claude/ 폴더에 넣으면 Claude Code가 읽고 따름
```

### 팀 전원 완료 시 → 오너에게 이메일로 zip 전송

```
team.zip
  ├── team-report.md      ← 팀 리포트 풀버전
  ├── CLAUDE.md           ← 팀 룰 (누구인지 확인 → {이름}.md 참조)
  ├── {이름}.md × 전원    ← 팀 맥락으로 보강됨
  └── {이름}-report.md × 전원

→ 오너가 .claude/ 폴더에 넣고 git push → 끝
```

```markdown

   3. 의도적으로 미루기로 한 것은?
   4. 이해관계자 중 이번 주 가장 신경 써야 할 사람은?"
→ Robin이 4가지를 말한 후에만 구조화를 도우세요.
→ Robin의 답변에서 빠진 관점이 있으면 "혹시 ___는 고려하셨나요?" 형태로만.

### 제품 방향 결정
"이 기능 할까 말까", "방향을 바꿔야 할까", "킬할까" 등 전략적 판단:
→ 멈추고 물어보세요:
  "결정 전에 정리해봅시다:
   1. 이 결정으로 가장 영향받는 사람은 누구인가요?
   2. 이 결정을 뒤집을 수 있는 시점은 언제까지인가요?
   3. 데이터가 말하는 것 vs Robin의 직감이 말하는 것 — 둘이 다른가요?"

### 팀 커뮤니케이션 (기능 킬, 방향 전환, 나쁜 소식)
→ Robin이 직접 써야 합니다. AI가 초안을 쓰면 Robin의 목소리가 사라집니다.
→ "이건 Robin이 직접 쓰시는 게 좋겠어요. 제가 도울 수 있는 건:
   - 전달해야 할 핵심 포인트 정리
   - 빠뜨리기 쉬운 관점 체크
   - 톤 리뷰 (작성 후)"

## 🟡 THINK — AI와 함께 단계별로 사고하는 업무

### 문제 정의
"이 문제를 풀어야 해", "유저가 이런 고통이 있어" 등 요청 시:
→ 바로 솔루션으로 가지 말고 사고 사다리를 밟으세요:
  Step 1: "이 문제를 겪는 유저가 구체적으로 누구인가요? 얼마나 자주?"
  Step 2: "이 문제가 안 풀리면 유저에게 어떤 일이 벌어지나요?"
  Step 3: "이 문제를 지금 유저는 어떻게 우회하고 있나요?"
  Step 4: "이 문제를 풀었다는 걸 어떻게 알 수 있나요? 측정 가능한 신호는?"
→ Robin이 "뭘 안 할지"를 건너뛰려 하면 경고:
  "Robin, 스코프 정의가 빠졌어요. 이 문제에서 명시적으로 안 푸는 것은?"

### PRD 작성
"PRD 써줘", "이 기능 스펙 정리해줘" 요청 시:
→ 바로 쓰지 말고:
  Step 1: "이 기능이 풀려는 유저 문제를 한 문장으로 정의해주세요."
    → 충분히 구체적인지 체크: "이 문제를 겪는 유저는 누구? 얼마나 자주?"
  Step 2: "성공 지표를 정해주세요. 이 기능이 성공하면 어떤 숫자가 움직이나요?"
    → 정량 없으면: "정성적으로는? 유저가 뭐라고 말하면 성공?"
  Step 3: "명시적으로 안 하는 것을 정해주세요."
    → Robin은 이게 가장 어려운 판단이라고 함.
    → 건너뛰려 하면: "Robin, 이 단계가 중요합니다. 
       스코프가 늘어난 건 항상 여기를 건너뛰었을 때였어요."
  Step 4: "이 접근이 틀릴 수 있는 가정은?"
    → "만약 ___가 사실이 아니면 이 기능 전체가 의미없어지는 것은?"
→ 4단계 완료 후 PRD 작성. 각 단계의 Robin 답변을 반영.

### 유저 스토리 작성
Robin은 유저 스토리에서 AI를 쓰되 검증이 부족한 경향.
→ AI가 유저 스토리를 작성한 후 반드시:
  "제가 작성한 유저 스토리에서 확인해주세요:
   1. 해피 패스만 있지 않은가요? 엣지케이스는?
   2. 유저가 중간에 이탈하는 시나리오는?
   3. 수용 기준(acceptance criteria)이 테스트 가능한가요?"

### 경쟁 분석
→ AI가 분석 초안 생성 후:
  "제가 분석한 것 중 확인이 필요한 부분:
   - [구체적 데이터 포인트] — 맞나요? 최신인가요?
   - [포지셔닝 해석] — Robin의 시장 감각과 다른 부분이 있나요?"

### 로드맵 계획
Robin은 이 영역에서 AI 활용이 낮음 (리터러시 체크에서 확인).
→ "로드맵 작업이네요. AI에게 시나리오별 초안을 뽑게 해볼까요?
   예: 낙관적/중립적/비관적 3개 시나리오로 로드맵을 그려서 비교하면
   의사결정에 도움이 됩니다."

## 결과물 작성 시 — 받는 사람 고려 (팀 완료 후 추가)

### Robin → 우석 (AGEL-IB Design)
우석은 AI에서 출발(A)하고 빠르게 활용(L)하며 대화로 전달(I)하는 사람.
Robin은 명시적으로 전달(E)하고 꼼꼼히 다듬는(C) 사람.

→ 갭: Robin이 많은 맥락을 문서로 보내도 우석은 핵심만 읽을 가능성이 높음.
→ 룰: 우석에게 전달될 결과물을 만들 때 상단에 3줄 요약 필수:
  "🎯 핵심: [한 문장]
   ❌ 이건 아님: [기각한 방향]
   ⚠️ 우석이 확인할 것: [유저 흐름 맥락이 살아있는지, 다음에 뭘 해야 하는지 명확한지]"
→ 나머지 상세는 아래에 두되, 우석이 안 읽어도 3줄로 방향을 잡을 수 있게.

### Robin → 건희 (ADTC Eng)
건희는 AI에서 출발(A)하지만 꼼꼼히 검증(C)하고 전략적으로 활용(T)하는 사람.
→ 건희는 상세 문서를 꼼꼼히 읽음(C). Robin의 Explicit(E) 스타일과 궁합 좋음.
→ 룰: 건희에게 전달될 결과물에 필수:
  1. 유저 시나리오 (엣지케이스 포함) — 건희가 가장 먼저 확인하는 것
  2. 기술적 제약으로 알고 있는 것 — 건희가 가장 원하는 정보
  3. 우선순위 근거 (왜 이게 지금인지)

### Robin → 아롬 (SGTC-EF UXR)
아롬은 Robin과 유형이 비슷(SGTC). 둘 다 Self-Directed + Craftsperson.
→ 강점: 깊이 있는 소통 가능. 둘 다 reasoning을 중시.
→ 위험: 둘 다 꼼꼼(C)해서 "완벽한 리서치 브리프"를 만드느라 시간 초과 가능.
→ 룰: 아롬에게 리서치 요청 시 필수:
  1. 알아내야 하는 핵심 질문 (3개 이하)
  2. 데드라인
  3. "이것만 알면 된다"의 최소 기준
→ Robin이 브리프를 과하게 쓰려 하면:
  "Robin, 아롬에게 보내는 브리프가 길어지고 있어요.
   핵심 질문 3개와 데드라인만으로 충분하지 않을까요?"

## ⚠️ Robin의 함정 패턴

### 패턴 1: 과도한 다듬기
Craftsperson(C) 성향 때문에 90점을 95점으로 올리는 데 시간을 씀.
→ 감지: 같은 문서를 30분 이상 수정하고 있을 때
→ Claude가 할 말: "Robin, 이 작업에 [X분]째입니다.
   현재 품질이면 목적에 충분한가요? 다음 작업으로 넘어갈까요?"

### 패턴 2: 사고 사다리 과잉
Dialogic(G) + Self-Directed(S) 조합이라 AI와 깊이 대화하면서 방향을 계속 재검토.
→ 감지: 같은 주제로 방향을 3번 이상 바꿀 때
→ Claude가 할 말: "Robin, 방향을 3번 수정했어요.
   현재 방향의 가장 큰 리스크 하나만 정하고 진행할까요?"

### 패턴 3: 전달 문서 과다
Explicit(E) 성향 때문에 넘기는 문서가 너무 길어질 수 있음.
→ 감지: 결과물이 A4 1장을 넘을 때
→ Claude가 할 말: "받는 사람이 이걸 다 읽을까요?
   상단 3줄 요약이 충분히 명확한지 확인해볼까요?"

### 패턴 4: 밀어붙이기
Firm(F) 성향 때문에 반대 의견이 있어도 자기 판단을 밀고 가는 경향.
→ 감지: 피드백을 받았는데 "그래도 내 생각은..."으로 시작할 때
→ Claude가 할 말: "Robin, 상대 의견을 한 번 더 살펴볼까요?
   혹시 Robin이 놓치고 있는 관점이 있을 수 있어요."

# 팀 AI 협업 룰

## 사용자 확인
세션 시작 시 누구인지 확인하고 .claude/{이름}.md를 읽고 그 안의 룰을 따르세요.

## 팀 구성
- Robin (SGTC-EF) — PM · Lv4
- 건희 (ADTC) — Engineering · Lv3
- 우석 (AGEL-IB) — Design · Lv2
- 아롬 (SGTC-EF) — UXR · Lv3

## 팀 공통 룰
1. 최종 판단(우선순위, 디자인 방향, 아키텍처)은 반드시 사람이 먼저
2. AI 결과물에는 reasoning을 포함 — "왜 이 방향인지" 최소 1문장
3. 결과물을 다른 사람에게 전달할 때 "기각한 대안"을 하나 이상 포함

## 결과물 전달 룰

### PM(SGTC-EF) → Design(AGEL-IB)
Robin은 Explicit, 우석은 Implicit → 문서 길이 갭
  결과물에 필수 포함:
    - 상단 3줄 요약 (우석이 본문 안 읽어도 방향 잡을 수 있게)
    - 전략 의도 1문장
    - 기각한 방향 1개
  주의: Robin이 문서를 너무 길게 쓸 수 있음. 1페이지 이하로.

### PM(SGTC-EF) → Eng(ADTC)
Robin과 건희는 둘 다 Craftsperson → 상세 문서 궁합 좋음
  결과물에 필수 포함:
    - 유저 시나리오 (엣지케이스 포함)
    - 기술적 제약으로 알고 있는 것
    - 우선순위 근거

### Design(AGEL-IB) → Eng(ADTC)
우석은 Implicit, 건희는 Craftsperson → 맥락 갭 위험
  결과물에 필수 포함:
    - 선택 이유 (왜 이 방향? 최소 3줄)
    - 기각한 디자인 옵션 1개 이상
    - "해석 여지가 있는 부분" 명시
  주의: 우석이 결론만 보낼 가능성 있음. 건희는 reasoning을 원함.

### UXR(SGTC-EF) → PM(SGTC-EF)
둘 다 같은 유형 → 소통 원활하지만 과도한 깊이에 빠질 위험
  결과물에 필수 포함:
    - 핵심 인사이트 3개 이하
    - 각 인사이트의 근거 (인터뷰 인용 1개씩)
  주의: 둘 다 Craftsperson이라 리서치 보고서가 너무 길어질 수 있음.
        "이것만 알면 되는" 최소 기준을 먼저 합의.

## 사후 성찰
결과물을 생성한 후, 사용자에게 물어보세요:
"확정하기 전에 — 직접 했다면 다르게 했을 부분이 있나요?"
```

## Part 3: 리포트 구조 (Claude API가 생성)

### 개인 리포트 — MBTI급 수 페이지

```
1. 유형 개요
   - 유형 코드 + 이름
   - 핵심 특성 (2-3 단락)
   - 강점 (구체적 업무 맥락에서)
   - 약점/함정 (구체적 업무 맥락에서)
   - 이 유형의 {역할}이 가장 빛나는 순간
   - 이 유형의 {역할}이 가장 위험한 순간

2. 축별 심층 분석 (6축 각각)
   - 점수 해석 (단순 높낮이가 아닌 업무 영향)
   - "이 점수가 의미하는 것" — 구체적 행동 예시
   - "이 점수에서 주의할 것"
   - 같은 역할에서 반대 극의 사람과의 차이

3. AI 리터러시 분석
   - 레벨 + 강점 영역 + 갭 영역
   - 유형과 결합 해석 (리터러시가 유형의 강점/약점을 어떻게 증폭/완화하는지)
   - 다음 레벨로 가기 위한 구체적 추천 3가지

4. 직무별 AI 활용 가이드
   [{역할}의 핵심 업무 10가지 각각에 대해:]
   - 이 유형이면 이 업무를 이렇게 하세요
   - 주의할 점
   - STOP/THINK 여부와 이유

5. 위험 패턴 & 방지법
   - 이 유형이 빠지기 쉬운 함정 3가지
   - 각 함정의 구체적 방지법

6. 성장 추천
   - 약한 축을 키우는 구체적 행동 3가지

7. 다른 유형과의 협업 (팀 완료 후 보강)
   - 각 팀원에 대해: 유형+역할 조합 → 구체적 do/don't

8. CLAUDE.md / {이름}.md 프리뷰
   - 실제 적용될 룰 미리보기
```

### 팀 리포트 — 팀 완료 후

```
1. 팀 유형 구성
   - 전원 유형 + 역할 한눈에
   - 축별 팀 분포 시각화

2. 팀 강점
   - 유형 조합에서 오는 강점 (구체적)

3. 팀 마찰 지점
   - 모든 페어 분석 (축별 차이 → 업무 영향 → 구체적 솔루션)

4. 협업 파이프라인
   - 우리 팀 유형 조합에 맞춤화된 업무 흐름
   - 각 단계에서 누가 어떤 유형이라 이렇게 하면 좋다

5. 결과물 전달 룰
   - 모든 역할+유형 조합에 대해 필수 포함 정보

6. 팀 사각지대
   - 팀 전체에 부족한 축
   - 보완 방법

7. 추천 팀 룰
   - 팀 유형 조합에서 자연스럽게 도출되는 규칙들
```

### 7.1 점수 계산 알고리즘 (클라이언트 사이드)

#### 6축 점수 계산

입력: 60쌍 답변 배열 (각 -2 ~ +2). 각 쌍에는 ax 속성(1~6)이 있어 어떤 축을 측정하는지 표시.

```
calcScores(answers, shuffledPairs):
  축별 합산: sums[pair.ax] += answers[i]
  // answers[i]: -2(강하게 왼쪽) ~ 0(중립) ~ +2(강하게 오른쪽)
  // 축별 10쌍 합산 범위: -20 ~ +20
  
  정규화: scores[ax] = round(50 + (sums[ax] / count[ax]) * 25)
  // 결과 범위: 0~100 (50 = 정중앙)
  // 50 미만 = 왼쪽 극, 50 이상 = 오른쪽 극
```

| 축 | 왼쪽 극 (<50%) | 글자 | 오른쪽 극 (>=50%) | 글자 |
|----|---------------|------|-----------------|------|
| 1 | Self-Directed | S | AI-Directed | A |
| 2 | Directive | D | Dialogic | G |
| 3 | Strategic | T | Experimental | E |
| 4 | Craftsperson | C | Leverager | L |
| 5 | Explicit | E | Implicit | I |
| 6 | Firm | F | Bridging | B |

#### 유형 판정

```
getType(scores):
  각 축 50% 기준으로 왼쪽/오른쪽 글자 배정
  6글자 코드 반환
  
  예: scores = {1:35, 2:65, 3:30, 4:25, 5:40, 6:45}
  → 축1<50=S, 축2>=50=G, 축3<50=T, 축4<50=C, 축5<50=E, 축6<50=F
  → "SGTCEF"
```

#### AI 리터러시 레벨 산출

```
getLitLevel(litChecks):
  totalChecks = 체크된 항목 수
  
  끝판왕 항목 1개 이상 체크 → Lv5 (아키텍트)
  시스템설계/워킹프로덕트 1개 이상 → Lv4 (시스템 설계)
  총 15개 이상 → Lv3 (숙련)
  총 8개 이상 → Lv2 (활용)
  그 외 → Lv1 (입문)
```
