# PM 사고 프레임워크

> 이 문서는 {이름}.md 생성 시 Claude API가 참조하는 레퍼런스입니다.
> PM 역할의 STOP/THINK 룰은 이 프레임워크에서 도출됩니다.
> 초안은 PM이 작성하고, 팀원들이 최종 컨펌합니다.

---

## 1. 문제를 대하는 방식

### 최고의 PM이 문제를 만나면

- 솔루션으로 가기 전에 **문제가 진짜인지** 먼저 검증한다
- "이 문제를 겪는 유저가 누구이고, 얼마나 자주 겪는지"를 데이터로 확인한다
- 문제의 크기를 측정한다 — 안 풀면 어떤 일이 벌어지는지
- 유저가 지금 이 문제를 어떻게 우회하고 있는지 파악한다 (현재 대안)

### 사고 단계

```
Step 1: 문제가 진짜인가?
  - 누가 겪는가? (구체적 유저 세그먼트)
  - 얼마나 자주? (빈도)
  - 얼마나 아픈가? (심각도)
  - 데이터가 있는가, 가설인가?

Step 2: 문제의 구조는?
  - 근본 원인 vs 증상을 구분했는가?
  - 이 문제가 다른 문제와 연결되어 있는가?
  - 풀면 다른 것도 풀리는가? (의존성)

Step 3: 풀 가치가 있는가?
  - 안 풀면 어떤 일이 벌어지는가?
  - 지금 유저는 어떻게 우회하고 있는가?
  - 우리가 풀어야 하는가, 아니면 시장이 풀 문제인가?
```

→ **STOP 룰 후보**: 문제 정의 없이 솔루션을 요청하면 멈추고 Step 1부터

### 반드시 사람이 확인할 것
- "이 문제가 진짜인지"의 최종 판단 — AI가 데이터를 분석해도 "풀 가치가 있는가"는 PM의 전략적 판단
- 유저의 감정적 맥락 — 데이터에 안 잡히는 것

## 2. 우선순위를 정하는 방식

### 최고의 PM이 우선순위를 정할 때

- 감이 아닌 프레임워크로 정하되, 프레임워크를 맹신하지 않는다
- "안 하는 것"을 먼저 정한다 — 하는 것보다 어렵고 중요하다
- One-way door(되돌릴 수 없는 결정)와 Two-way door(쉽게 바꿀 수 있는 결정)를 구분한다

### 사고 단계

```
Step 1: 이번에 반드시 달성해야 하는 한 가지 목표는?
  - 팀 전체가 동의하는 목표인가?
  - 측정 가능한가?

Step 2: 후보를 평가한다
  - Reach: 얼마나 많은 유저에게 영향?
  - Impact: 한 유저에게 얼마나 큰 영향?
  - Confidence: 이 추정을 얼마나 믿을 수 있는가?
  - Effort: 얼마나 걸리는가?
  - (RICE 또는 팀의 평가 프레임워크 사용)

Step 3: "안 하는 것"을 명시한다
  - 이번에 의도적으로 미루는 것은?
  - 미루는 이유는?
  - 언제 다시 볼 것인가?

Step 4: 되돌릴 수 있는가?
  - One-way door → 신중하게, 데이터 더 모으기
  - Two-way door → 빠르게 결정, 실행하면서 배우기
```

→ **STOP 룰 후보**: 우선순위 요청 시 AI가 먼저 추천하지 말고 PM의 판단을 먼저 듣기

### 반드시 사람이 확인할 것
- "안 하는 것" 결정 — AI는 모든 걸 하자고 할 수 있음
- 팀의 에너지/번아웃 수준 — 데이터에 없는 것
- 이해관계자의 정치적 맥락

## 3. PRD를 쓰는 방식

### 최고의 PM이 PRD를 쓸 때

- 솔루션이 아닌 문제부터 쓴다 (Amazon Working Backwards)
- "이 기능이 성공하면 세상이 어떻게 달라지는가?"를 먼저 정의한다
- 엣지케이스를 처음부터 고려한다 — 해피 패스만 쓰면 절반

### 사고 단계

```
Step 1: 프레스 릴리즈 (Working Backwards)
  - 이 기능이 완성되면 유저에게 어떤 가치를 주는가?
  - 한 문장으로 설명할 수 있는가?

Step 2: 성공 지표
  - 어떤 숫자가 움직여야 성공인가?
  - 정량이 없으면: 유저가 뭐라고 말하면 성공인가?

Step 3: 스코프
  - 명시적으로 안 하는 것은?
  - 이번 버전의 최소 범위는?
  - "이것만 있으면 가치를 전달할 수 있다"는 기준은?

Step 4: 리스크
  - 이 접근이 틀릴 수 있는 가정은?
  - 가정이 틀리면 어떤 일이 벌어지는가?
  - 가정을 검증할 가장 빠른 방법은?

Step 5: 유저 시나리오
  - 해피 패스
  - 엣지케이스 (기존 유저, 이탈, 에러, 권한)
  - 각 시나리오의 수용 기준
```

→ **THINK 룰 후보**: PRD 요청 시 Step 1~4를 순서대로 밟기

### 반드시 사람이 확인할 것
- "안 하는 것" — AI는 스코프를 넓히는 방향으로 갈 수 있음
- 유저 시나리오의 현실성 — AI가 만든 시나리오가 실제 유저 행동과 다를 수 있음
- 기술적 제약과의 정합성 — Eng과 대화해야 알 수 있는 것

## 4. 유저 스토리를 쓰는 방식

### 사고 단계

```
Step 1: JTBD 형식으로 시작
  "When ___, I want to ___, so I can ___"

Step 2: 수용 기준 (Acceptance Criteria)
  - Given [상황] When [행동] Then [결과]
  - 테스트 가능해야 한다

Step 3: 엣지케이스 체크
  - 기존 유저가 이 기능을 처음 만나면?
  - 중간에 이탈하면?
  - 에러가 나면?
  - 권한이 없으면?
  - 데이터가 없으면 (빈 상태)?

Step 4: 의존성
  - 이 스토리를 완성하려면 먼저 뭐가 있어야 하는가?
  - 다른 스토리와 순서 의존이 있는가?
```

### 반드시 사람이 확인할 것
- AI가 만든 유저 스토리의 엣지케이스가 실제 유저 행동을 반영하는지
- 수용 기준이 실제로 테스트 가능한지

## 5. 의사결정을 커뮤니케이션하는 방식

### 최고의 PM이 결정을 전달할 때

- "뭘 결정했는가"보다 "왜 그렇게 결정했는가"가 더 중요하다
- 나쁜 소식일수록 직접, 빠르게, 솔직하게
- 팀의 노력을 인정한 후 방향을 말한다

### 사고 단계

```
Step 1: 누구에게 영향을 주는가?
  - 직접 영향: 작업이 바뀌는 사람
  - 간접 영향: 계획이 바뀌는 사람

Step 2: 메시지 구조
  - 결정 사항 (한 문장)
  - 왜 이 결정인가 (데이터 + 전략적 판단)
  - 팀의 기존 작업은 어떻게 되는가
  - 다음 스텝은 무엇인가

Step 3: 톤
  - 나쁜 소식: 팀의 노력 인정 → 결정 → 이유 → 다음 스텝
  - 방향 전환: 새 방향의 기회 → 전환 이유 → 기존 작업의 가치
```

→ **STOP 룰 후보**: 팀 커뮤니케이션(나쁜 소식)은 PM이 직접 작성. AI는 톤 리뷰만.

### 반드시 사람이 확인할 것
- 톤 — AI가 쓰면 "기업 공식 발표"처럼 됨. PM의 목소리가 있어야 함
- 팀원 개인에 대한 배려 — AI는 모름

## 6. 경쟁 분석하는 방식

### 사고 단계

```
Step 1: 경쟁의 범위 정의
  - 직접 경쟁 (같은 문제를 같은 방식으로 풂)
  - 간접 경쟁 (같은 문제를 다른 방식으로 풂)
  - 대체재 (유저가 지금 쓰는 우회 방법)

Step 2: 비교 축 정의
  - 유저 관점: 가격, UX, 기능, 신뢰
  - 비즈니스 관점: TAM, 성장률, 펀딩
  - 기술 관점: 스택, API, 확장성

Step 3: 포지셔닝
  - 우리가 이기는 축은?
  - 우리가 지는 축은?
  - 지는 축에서 싸울 것인가, 피할 것인가?
```

### 반드시 사람이 확인할 것
- AI가 만든 경쟁 데이터의 정확성 — 특히 가격, 최신 기능, 시장 규모
- 포지셔닝 판단 — "어디서 싸울 것인가"는 전략적 판단

## 7. 핸드오프하는 방식

### 최고의 PM이 결과물을 넘길 때

- 받는 사람이 "왜?"를 묻지 않아도 되게 만든다
- 받는 사람의 관점에서 필요한 정보를 넣는다

### 받는 역할별 필수 정보

```
PM → Design:
  - 유저 문제 (JTBD 형식)
  - 전략적 의도 (왜 이 방향)
  - 기각한 대안과 이유
  - 제약 조건 (기술적, 비즈니스적)
  - "해석 여지가 있는 부분" 명시

PM → Engineering:
  - 유저 시나리오 (해피 패스 + 엣지케이스)
  - 우선순위 근거 (왜 이게 지금)
  - 기술적 제약으로 알고 있는 것
  - 수용 기준 (테스트 가능한 형태)

PM → UXR:
  - 알아내야 하는 핵심 질문 (3개 이하)
  - 데드라인
  - "이것만 알면 된다"의 최소 기준
```

## 8. AI를 쓰는 방식

### 최고의 PM이 AI를 쓸 때

- AI를 "해줘" 도구가 아닌 "사고의 동반자"로 쓴다
- AI에게 시키기 전에 자기 판단을 먼저 세운다
- AI 결과를 그대로 쓰지 않고, 자기 기준으로 다듬는다
- AI가 놓치는 것을 안다: 정치적 맥락, 팀의 감정, 유저의 뉘앙스

### AI에게 맡겨도 되는 것
- 초안 작성 (PRD, 경쟁 분석, 미팅 요약)
- 데이터 정리/분석
- 엣지케이스 브레인스토밍
- 문서 포맷팅/구조화

### 반드시 사람이 해야 하는 것
- 우선순위 최종 결정
- "안 하는 것" 정의
- 팀 커뮤니케이션 (특히 나쁜 소식)
- 유저 문제의 진위 판단
- 전략적 방향 결정
