# Design 사고 프레임워크

> 초안은 PM이 작성하고, 디자이너가 최종 컨펌합니다.

---

## 1. 문제를 발견하는 방식

### 최고의 디자이너가 문제를 발견할 때

- "이걸 만들어달라"가 아닌 "이걸 왜 만들어야 하는가?"부터 시작한다
- 유저의 말이 아닌 유저의 행동을 관찰한다
- 문제의 범위를 넓혔다가(Discover) 좁히고(Define), 솔루션을 넓혔다가(Develop) 좁힌다(Deliver) — Double Diamond

### 사고 단계

```
Step 1: 진짜 문제 찾기 (Discover)
  - PRD에 적힌 문제가 진짜 문제인가?
  - 유저가 실제로 겪는 순간을 구체적으로 떠올릴 수 있는가?
  - 현재 유저가 어떻게 우회하고 있는가?

Step 2: 문제 정의 (Define)
  - 풀어야 할 문제를 한 문장으로
  - 이 문제를 풀면 유저의 삶이 어떻게 달라지는가?
  - 이 문제의 범위는? (뭘 포함하고 뭘 뺄 것인가?)

Step 3: 유저 맥락 확인
  - 이 화면에 도달하기 전에 유저가 한 일은?
  - 이 화면을 떠난 후 유저가 할 일은?
  - 유저의 감정 상태는? (급한가? 불안한가? 탐색 중인가?)
```

→ **STOP 룰 후보**: 와이어프레임을 그리기 전에 유저 맥락을 먼저 확인

### 반드시 사람이 확인할 것
- "유저가 생각 안 해도 되는 상태"인지 — AI는 기능적 완성도는 체크하지만 인지 부하를 못 봄
- 유저의 감정적 맥락 — 데이터에 없는 것

---

## 2. 디자인 방향을 결정하는 방식

### 최고의 디자이너가 방향을 정할 때

- 최소 3개의 방향을 탐색한 후 선택한다
- "왜 이 방향인지"를 기각한 대안과 함께 설명할 수 있다
- 직감으로 판단하되, 직감의 근거를 설명할 수 있다

### 사고 단계

```
Step 1: 최소 3개 방향 탐색
  - 완전히 다른 접근법 3개
  - 각각의 장단점
  - 각각이 유저의 어떤 측면을 강조하는지

Step 2: 평가 기준 정의
  - 유저 목표 달성까지의 단계 수
  - 인지 부하 (유저가 생각해야 하는 양)
  - 기존 경험과의 일관성
  - 기술적 실현 가능성

Step 3: 선택 + 근거
  - 선택한 방향과 그 이유
  - 기각한 방향과 기각 이유
  - "이 선택이 틀릴 수 있는 상황"
```

→ **STOP 룰 후보**: 디자인 방향을 정할 때 "왜 이 방향?" 먼저

### 반드시 사람이 확인할 것
- "느낌"으로 아는 것 — "기술적으로 맞지만 뭔가 이상한" 디자인
- 브랜드 톤과의 일관성 — AI가 기능적으로는 맞지만 브랜드 감성을 놓칠 수 있음

---

## 3. 와이어프레임/프로토타입을 만드는 방식

### 사고 단계

```
Step 1: 정보 구조
  - 이 화면에서 가장 중요한 정보는?
  - 유저가 가장 먼저 봐야 하는 것은?
  - 정보의 위계 (제목 → 핵심 → 상세 → 부가)

Step 2: 유저 플로우
  - 시작부터 목표 달성까지의 단계
  - 각 단계에서 유저가 해야 하는 행동
  - 분기점: 어디서 유저가 갈리는가?
  - 이탈점: 어디서 유저가 포기할 수 있는가?

Step 3: 상태 고려
  - 빈 상태 (데이터 없을 때)
  - 로딩 상태
  - 에러 상태
  - 성공 상태
  - 엣지: 데이터가 너무 많을 때, 너무 길 때

Step 4: 인터랙션
  - 화면 전환 방식 (어디서 어디로, 어떻게)
  - 피드백 (유저가 행동했을 때 시스템이 어떻게 반응)
  - 애니메이션의 목적 (장식이 아닌 맥락 전달)
```

### 반드시 사람이 확인할 것
- 빈 상태와 에러 상태 — AI가 해피 패스만 디자인하는 경향
- "유저가 다음에 뭘 해야 하는지 고민 안 해도 되는가?"

---

## 4. 디자인 시스템을 관리하는 방식

### 사고 단계

```
Step 1: 재사용 가능한가?
  - 기존 컴포넌트로 해결 가능한가?
  - 새 컴포넌트가 필요하다면, 다른 곳에서도 쓸 수 있는가?

Step 2: 일관성
  - 기존 패턴과 같은 규칙을 따르는가?
  - 같은 문제를 다른 방식으로 풀고 있지 않은가?

Step 3: 확장성
  - 데이터가 바뀌면? (짧은 텍스트 vs 긴 텍스트)
  - 언어가 바뀌면?
  - 디바이스가 바뀌면?
```

### 반드시 사람이 확인할 것
- 시스템 철학과의 일관성 — AI는 개별 컴포넌트는 잘 만들지만 전체 시스템의 "왜"를 모름

## 5. 사용성을 평가하는 방식

### Nielsen 10 Heuristics 기반 체크

```
1. 시스템 상태 가시성 — 유저가 지금 어디인지 아는가?
2. 현실 세계와의 매칭 — 유저의 언어를 쓰고 있는가?
3. 유저 제어 + 자유 — 실수를 되돌릴 수 있는가?
4. 일관성 + 표준 — 같은 것은 같아 보이는가?
5. 에러 방지 — 실수하기 어려운 구조인가?
6. 기억보다 인식 — 유저가 기억할 필요 없이 보고 판단할 수 있는가?
7. 유연성 + 효율 — 초보자와 전문가 모두 사용 가능한가?
8. 미니멀 디자인 — 불필요한 정보가 없는가?
9. 에러 회복 — 에러 메시지가 해결 방법을 알려주는가?
10. 도움말 — 필요할 때 도움을 찾을 수 있는가?
```

### 반드시 사람이 확인할 것
- 접근성 (WCAG 2.1 AA) — 자동 검사 도구 + 수동 스크린리더 테스트
- 실제 유저 맥락에서의 사용성 — 자동 분석만으로는 부족

---

## 6. 핸드오프하는 방식

### 받는 역할별 필수 정보

```
Design → Engineering:
  - 컴포넌트별 상태 정의 (default, hover, focus, disabled, error, loading, success)
  - 인터랙션 설명 ("이 버튼을 누르면 → 이 모달이 → 300ms 후 닫힘")
  - "왜 이 방향인지" — 선택 이유 + 기각 옵션
  - "해석 여지가 있는 부분" 명시
  - 반응형 동작

Design → PM:
  - 디자인 결정의 근거
  - 유저 플로우상 고려한 것과 못한 것
  - 기술적 제약으로 타협한 부분
```

### 반드시 사람이 확인할 것
- Eng이 "이 디자인을 어떻게 해석할지" — 디자이너의 의도와 Eng의 해석 사이 갭

---

## 7. AI를 쓰는 방식

### AI에게 맡겨도 되는 것
- 아이디에이션 (다양한 방향 탐색)
- 와이어프레임 초안
- UI 카피
- 에셋 생성 (아이콘, 일러스트)
- 접근성 체크 초안
- 경쟁사 UI 분석

### 반드시 사람이 해야 하는 것
- 디자인 방향 최종 결정 — "왜 이 방향인지"
- "유저가 생각 안 해도 되는 상태인지" 판단
- 브랜드 톤 + 감성 판단
- 빈 상태/에러 상태의 UX 설계
- 핸드오프 시 "해석 여지" 명시
