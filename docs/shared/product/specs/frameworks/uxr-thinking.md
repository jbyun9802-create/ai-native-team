# UXR 사고 프레임워크

> 초안은 PM이 작성하고, UX 리서처가 최종 컨펌합니다.

---

## 1. 리서치를 설계하는 방식

### 최고의 리서처가 리서치를 설계할 때

- "뭘 알아내야 하는가?"를 먼저 정의한다 — 방법은 그 다음
- 리서치 질문과 비즈니스 질문을 구분한다
- "이 리서치가 어떤 결정을 바꿀 수 있는가?"를 먼저 확인한다

### 사고 단계

```
Step 1: 리서치 질문 정의
  - 알아내야 하는 것을 질문 형태로 (최대 3개)
  - 이 질문에 답하면 어떤 결정이 가능해지는가?
  - 이미 알고 있는 것은 뭔가? (기존 데이터 확인)

Step 2: 방법 선택
  - 탐색적 (뭘 모르는지 모를 때) → 인터뷰, 관찰
  - 평가적 (가설이 있을 때) → 사용성 테스트, 설문
  - 정량 vs 정성: "얼마나?"는 정량, "왜?"는 정성

Step 3: 참여자 설계
  - 누구에게 물어야 하는가? (대표성)
  - 몇 명이면 충분한가?
  - 리크루팅의 현실적 제약

Step 4: 윤리 확인
  - 동의 범위가 명확한가?
  - 취약한 참여자가 있는가?
  - 데이터를 어떻게 보관/폐기할 것인가?
```

→ **STOP 룰 후보**: 리서치 설계 시 "이 리서치가 어떤 결정을 바꾸는가?" 먼저

### 반드시 사람이 확인할 것
- 리서치 질문의 방향성 — AI가 "재밌는 질문"을 만들 수 있지만 비즈니스와 연결되지 않을 수 있음
- 참여자 선정의 대표성 — AI는 이상적 참여자를 추천하지만 현실적 리크루팅 가능성은 다름
- 윤리적 판단 — AI에게 위임 불가

---

## 2. 인터뷰하는 방식

### 최고의 리서처가 인터뷰할 때

- 준비한 질문 목록을 읽는 게 아니라 대화를 한다
- "왜?"를 최소 3번 파고든다
- 유도 질문을 피한다 — "좋았나요?"가 아닌 "어땠나요?"
- 참여자의 비언어적 신호를 읽는다

### 사고 단계

```
Step 1: 가이드 작성
  - 핵심 질문 5-7개 (너무 많으면 깊이가 없어짐)
  - 각 질문의 프로빙 방향 ("왜?", "구체적으로?", "예를 들면?")
  - 유도 질문 체크: "~하지 않나요?"를 "어떤가요?"로

Step 2: 인터뷰 중
  - 참여자가 말한 것 vs 하는 것의 차이를 관찰
  - 감정적 반응이 나오는 순간을 캐치
  - 예상과 다른 답이 나오면 → 더 파고들기 (확인 편향 방지)

Step 3: 인터뷰 직후
  - 5분 내로 핵심 인상/인사이트를 메모
  - "나를 가장 놀라게 한 것"
  - "내 가설을 지지한 것 / 반박한 것"
```

### 반드시 사람이 확인할 것
- 비언어적 신호 — 머뭇거림, 표정, 톤. AI 전사에 안 잡힘
- 인터뷰 중 방향 전환 판단 — "이 주제를 더 파야 하는가?"

---

## 3. 데이터를 분석하는 방식

### 최고의 리서처가 정성 데이터를 분석할 때

- 결론을 먼저 정하고 증거를 찾지 않는다 (확인 편향 방지)
- 체계적 프로세스를 따른다 — Thematic Analysis 6단계
- 반대 증거를 의도적으로 찾는다

### Thematic Analysis 6단계 (Braun & Clarke)

```
1. 데이터 익숙해지기
   - 전사본 전체를 읽으며 메모
   - AI가 할 수 있지만, 최소 3건은 직접 읽기

2. 초기 코드 생성
   - 데이터에서 의미 있는 단위 추출
   - 코드는 데이터에서 나와야 함 (선입견 아닌)

3. 테마 탐색
   - 코드를 그룹핑하여 상위 테마 도출
   - "이 코드들이 왜 같이 묶이는가?"

4. 테마 검토
   - 각 테마가 데이터로 충분히 지지되는가?
   - 테마 간 겹침이나 빈 곳은?

5. 테마 정의 + 네이밍
   - 각 테마의 핵심을 한 문장으로
   - 테마 이름이 내용을 정확히 대표하는가?

6. 보고서 작성
   - 테마 → 인사이트 → 행동 추천
   - 각 인사이트에 인용구 근거
```

→ **THINK 룰 후보**: 분석 시 6단계를 건너뛰지 않기

### 반드시 사람이 확인할 것
- 확인 편향 체크 — "내가 원하는 결론을 지지하는 데이터만 보고 있지 않은가?"
- AI 분석과 직접 분석의 차이 — 차이가 나는 곳이 가장 중요한 인사이트
- 인용구의 맥락 — AI가 뽑은 인용구가 원래 맥락에서 같은 의미인지

## 4. 편향을 감지하는 방식

### 체크리스트

```
리서치 설계 단계:
  □ 참여자 선정이 편향되지 않았는가? (찾기 쉬운 사람만 아닌가?)
  □ 질문이 특정 답을 유도하지 않는가?
  □ 팀이 원하는 결과가 있고, 그게 질문 설계에 영향을 주지 않았는가?

분석 단계:
  □ 다수 의견만 보고 있지 않은가? 소수 의견도 분석했는가?
  □ 예상과 다른 데이터를 무시하고 있지 않은가?
  □ AI 분석 결과를 검증 없이 수용하고 있지 않은가?

보고 단계:
  □ 불리한 발견도 포함했는가?
  □ 불확실성을 명시했는가?
  □ 8명의 인터뷰로 일반화하고 있지 않은가?
```

### 반드시 사람이 확인할 것
- 내 편향 — "이 결과가 나왔으면 좋겠다"는 마음이 분석에 영향을 주지 않았는가?
- AI 편향 — AI도 학습 데이터의 편향이 있음. AI 결과를 절대적으로 믿지 않기

---

## 5. 인사이트를 도출하는 방식

### 최고의 리서처가 인사이트를 도출할 때

- "데이터가 말하는 것"과 "내가 해석하는 것"을 구분한다
- "So what?" 테스트: "그래서 뭘 해야 하는데?"에 답할 수 있어야 인사이트

### 사고 단계

```
Step 1: 발견 (Finding)
  - 데이터에서 본 것 그대로
  - "8명 중 5명이 X를 했다"

Step 2: 인사이트 (Insight)
  - 발견의 의미 해석
  - "유저들이 X를 하는 이유는 Y 때문이다"

Step 3: 추천 (Recommendation)
  - 인사이트에 기반한 행동
  - "따라서 Z를 해야 한다"
  - 추천의 확신 수준: 높음/중간/낮음

Step 4: "So what?" 테스트
  - 이 인사이트가 없었으면 어떤 결정이 달라졌을까?
  - 이 인사이트로 팀이 구체적으로 뭘 할 수 있는가?
  - 할 수 없다면 → 인사이트가 아니라 관찰
```

### 반드시 사람이 확인할 것
- 인사이트의 실행 가능성 — "흥미로운 발견"과 "팀이 행동할 수 있는 인사이트"는 다름
- 근거의 강도 — 인터뷰 3건에서 나온 것과 설문 200건에서 나온 것의 무게가 다름

## 6. 보고서를 쓰는 방식

### 사고 단계

```
Step 1: 독자 파악
  - 누가 읽는가? (PM? 경영진? 전체 팀?)
  - 10분 안에 읽을 수 있는가?
  - 어떤 결정에 쓰일 것인가?

Step 2: 구조 (Pyramid Principle)
  - 결론 먼저 → 근거 → 상세
  - 핵심 인사이트 3개 이하
  - 각 인사이트에 인용구 + 추천

Step 3: 불확실성 명시
  - 확신 수준 (높음/중간/낮음)
  - 한계 (참여자 수, 대표성, 방법의 한계)
  - "추가 리서치가 필요한 것"
```

### 반드시 사람이 확인할 것
- "깔끔한 이야기"에 빠지지 않기 — 현실은 지저분함. AI는 깔끔하게 정리하려는 경향
- 불리한 발견도 포함했는지 — 이해관계자가 듣고 싶지 않은 것

---

## 7. 핸드오프하는 방식

```
UXR → PM:
  - 핵심 인사이트 (3개 이하)
  - 각 인사이트의 근거 (인용구 1-2개씩)
  - 추천 행동 + 확신 수준
  - 한계 + 추가 리서치 필요 여부

UXR → Design:
  - 유저 행동 패턴 (실제 관찰)
  - 유저가 쓰는 언어 (UI 카피 참고)
  - 유저가 기대하는 흐름 (멘탈 모델)
```

## 8. AI를 쓰는 방식

### AI에게 맡겨도 되는 것
- 트랜스크립션
- 초기 코드 생성 (검토 필수)
- 대량 데이터 분류 (NPS, 리뷰 등)
- 보고서 초안
- "반대 해석" 생성 (편향 체크용)

### 반드시 사람이 해야 하는 것
- 리서치 질문 정의 — "뭘 알아내야 하는가?"
- 인터뷰 중 방향 전환 판단
- 확인 편향 체크 — AI도 편향됨
- 인사이트의 실행 가능성 판단
- 윤리적 판단 (동의, 취약 참여자)
- "8명으로 일반화해도 되는가?"의 판단
