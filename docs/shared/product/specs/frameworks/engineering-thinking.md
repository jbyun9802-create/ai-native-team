# Engineering 사고 프레임워크

> 이 문서는 {이름}.md 생성 시 Claude API가 참조하는 레퍼런스입니다.
> 초안은 PM이 작성하고, 엔지니어가 최종 컨펌합니다.

---

## 1. 아키텍처 결정을 대하는 방식

### 최고의 엔지니어가 아키텍처를 결정할 때

- 트레이드오프를 명시적으로 적는다 — "이걸 선택하면 이걸 포기한다"
- 결정의 이유를 ADR(Architecture Decision Record)로 남긴다
- "지금 필요한 것"과 "미래에 필요할 것"을 구분한다 (YAGNI)
- 되돌리기 어려운 결정과 쉬운 결정을 구분한다

### 사고 단계

```
Step 1: 요구사항 명확화
  - 지금 필요한 것은 정확히 무엇인가?
  - 6개월 뒤 예상되는 스케일/요구사항은?
  - 비기능 요구사항: 성능, 가용성, 보안, 비용

Step 2: 옵션 나열 + 트레이드오프
  - 최소 3개 옵션을 검토했는가?
  - 각 옵션의 장단점을 표로 정리
  - "이 옵션을 선택하면 포기하는 것"을 명시

Step 3: 팀 맥락 확인
  - 우리 팀이 운영해본 경험이 있는가?
  - 온콜 로테이션/배포 파이프라인에 맞는가?
  - 팀원들이 유지보수할 수 있는가?

Step 4: 결정 + 기록
  - ADR 작성: 맥락, 결정, 이유, 결과
  - "이 결정을 재검토해야 하는 시점"을 명시
```

→ **STOP 룰 후보**: 아키텍처 결정 요청 시 AI가 바로 추천하지 말고 트레이드오프 정리부터

### 반드시 사람이 확인할 것
- 팀의 운영 역량과의 매칭 — AI는 "이론적 최선"을 추천하지만 우리 팀이 운영 가능한지는 모름
- 비용 임팩트 — 클라우드 비용, 라이센스 비용
- 보안 함의 — 데이터 흐름, 인증/인가

## 2. 코드를 작성하는 방식

### 최고의 엔지니어가 코드를 쓸 때

- "동작하는 코드"가 아닌 "읽을 수 있는 코드"를 쓴다
- 다음 사람이 이 코드를 보고 "왜 이렇게 했는지" 알 수 있게 만든다
- AI가 생성한 코드도 자기가 쓴 것처럼 책임진다

### 사고 단계

```
Step 1: 가장 단순한 구현은?
  - 이 문제를 풀 수 있는 가장 간단한 방법은?
  - 과도한 추상화를 하고 있지는 않은가?

Step 2: 엣지케이스
  - 입력이 비어있으면?
  - 동시에 여러 요청이 오면?
  - 네트워크가 끊기면?
  - 데이터가 예상과 다른 형식이면?

Step 3: 다음 사람을 위한 코드
  - 변수명/함수명이 의도를 설명하는가?
  - 복잡한 로직에 "왜"를 설명하는 주석이 있는가?
  - 6개월 뒤 처음 보는 사람이 이해할 수 있는가?

Step 4: 테스트
  - 이 코드가 깨지는 경우를 알고 있는가?
  - 그 경우를 테스트했는가?
```

### 반드시 사람이 확인할 것
- AI 생성 코드의 아키텍처 적합성 — 동작은 하지만 팀 패턴과 다를 수 있음
- 보안 취약점 — AI는 동작에 집중하고 보안을 놓칠 수 있음
- 에러 핸들링의 적절성 — AI가 에러를 삼키는 패턴을 자주 생성

---

## 3. 코드 리뷰하는 방식

### 최고의 엔지니어가 리뷰할 때

- "동작하는가?"보다 "유지보수 가능한가?"를 본다
- AI가 생성한 코드인지 여부와 무관하게 같은 기준으로 리뷰한다
- 작성자에게 "왜 이렇게 했는지" 물어보는 것을 두려워하지 않는다

### 사고 단계

```
Step 1: 전체 그림
  - 이 변경이 무엇을 풀려고 하는가?
  - 접근 방식이 적절한가?
  - 더 단순한 방법은 없는가?

Step 2: 위험 지점
  - 가장 복잡한 부분은 어디인가?
  - 가장 깨지기 쉬운 부분은?
  - 성능에 영향을 주는 부분은?

Step 3: AI 코드 특유의 체크
  - 팀의 기존 패턴과 다른 패턴을 쓰고 있지 않은가?
  - 불필요한 복잡성이 있지 않은가? (AI가 과잉 설계하는 경향)
  - 에러를 삼키고 있지 않은가?
  - N+1 쿼리, 연결 풀 누수 같은 숨은 문제는?

Step 4: 피드백
  - "고쳐라"가 아닌 "이건 왜 이렇게 했어?"로 시작
  - 칭찬할 것은 칭찬 — 좋은 패턴은 팀에 공유
```

→ **THINK 룰 후보**: 코드 리뷰 시 Step 1~3을 순서대로

### 반드시 사람이 확인할 것
- 비즈니스 로직의 정확성 — AI는 코드 구조는 보지만 비즈니스 의도와 맞는지는 모름
- 팀 컨벤션 — 기존 코드베이스의 패턴과 일관성

## 4. 디버깅하는 방식

### 최고의 엔지니어가 디버깅할 때

- 감이 아닌 가설로 접근한다
- 한 번에 하나씩 변수를 바꾼다
- 최근에 바뀐 것을 먼저 의심한다

### 사고 단계

```
Step 1: 증상 정리
  - 정확히 무엇이 잘못되었는가? (기대 vs 실제)
  - 언제부터? (최근 배포, 특정 시간대?)
  - 얼마나 자주? (항상? 간헐적?)
  - 어디서? (특정 엔드포인트? 특정 유저?)

Step 2: 가설 세우기
  - 가장 가능성 높은 원인 3개
  - 각 가설을 검증할 방법
  - 가장 빠르게 검증할 수 있는 가설부터

Step 3: 검증
  - 한 번에 하나의 가설만 검증
  - 검증 결과를 기록
  - 가설이 틀리면 다음으로, 맞으면 깊이 들어가기

Step 4: 수정 + 회고
  - 수정 후 문제가 재현되지 않는지 확인
  - "이 버그가 다시 나오지 않으려면?" → 테스트 추가
  - "이 유형의 버그를 미리 잡으려면?" → 모니터링/경고 추가
```

### 반드시 사람이 확인할 것
- 간헐적 버그의 재현 조건 — AI 로그 분석만으로는 충분하지 않을 수 있음
- 수정의 부작용 — "이걸 고치면 저게 깨지지 않는가?"

## 5. 테스트 전략 세우는 방식

### 사고 단계

```
Step 1: 뭘 테스트할 것인가?
  - 이 코드에서 가장 깨지기 쉬운 부분은?
  - 유저에게 가장 큰 영향을 주는 실패는?
  - AI가 생성한 코드에서 특히 주의할 것: 에러 핸들링, 경계값

Step 2: 어떻게 테스트할 것인가?
  - 단위 테스트: 함수/메서드 레벨 (빠르고 많이)
  - 통합 테스트: 컴포넌트 간 상호작용 (핵심 경로만)
  - E2E 테스트: 유저 시나리오 (가장 중요한 것만)

Step 3: 테스트의 품질 확인
  - 이 테스트가 실제로 무언가를 검증하는가?
  - "코드가 하는 일"을 테스트하는 게 아니라 "코드가 해야 하는 일"을 테스트하는가?
  - 테스트가 깨지면 실제 문제인가, 테스트 자체의 문제인가?
```

### 반드시 사람이 확인할 것
- AI가 생성한 테스트의 의미 — "테스트가 있다"와 "의미있는 테스트가 있다"는 다름
- 엣지케이스 커버리지 — AI는 해피 패스 테스트에 편향

## 6. 기술 부채를 대하는 방식

### 사고 단계

```
Step 1: 이게 부채인가, 의도적 단순화인가?
  - 당시에 그렇게 한 이유가 있었는가?
  - 그 이유가 지금도 유효한가?

Step 2: 비용 평가
  - 이 부채 때문에 새 기능 개발이 얼마나 느려지는가?
  - 이 부채 때문에 버그가 발생하는 빈도는?
  - 리팩토링 비용 vs 유지 비용

Step 3: 지금 갚을 것인가?
  - 지금 리팩토링하면 현재 스프린트에 영향은?
  - 안 갚으면 6개월 뒤 비용은?
  - 점진적으로 갚을 수 있는가, 한번에 해야 하는가?
```

### 반드시 사람이 확인할 것
- 리팩토링의 범위 — AI는 "이상적 구조"를 추천하지만 현실적 제약을 모름
- 팀의 현재 여력 — 기술 부채 상환은 비즈니스 우선순위와 균형

## 7. 핸드오프하는 방식

### 받는 역할별 필수 정보

```
Eng → PM:
  - 기술적 제약 (이건 안 되거나 비쌈)
  - 가능한 대안
  - 예상 소요 시간 (낙관/중립/비관)

Eng → Design:
  - 기술적 가능/불가능
  - 구현 비용이 크게 다른 디자인 옵션이 있다면 미리 공유
  - 기존 컴포넌트로 커버 가능한 것 vs 새로 만들어야 하는 것

Eng → 다음 Eng (코드 리뷰/인수인계):
  - 아키텍처 결정과 이유 (ADR)
  - "이 코드에서 가장 주의할 부분"
  - 알려진 한계/부채
```

## 8. AI를 쓰는 방식

### 최고의 엔지니어가 AI를 쓸 때

- AI를 "코드 생성기"가 아닌 "페어 프로그래밍 파트너"로 쓴다
- AI가 생성한 코드를 라인별로 읽고 이해한 후에만 커밋한다
- "AI가 만들었으니까"로 책임을 전가하지 않는다

### AI에게 맡겨도 되는 것
- 보일러플레이트 코드 생성
- 테스트 코드 초안
- 기술 문서 초안
- 에러 메시지 기반 디버깅 힌트

### 반드시 사람이 해야 하는 것
- 아키텍처 결정
- 보안 리뷰
- AI 코드의 성능/에러 핸들링 검증
- 비즈니스 로직의 정확성 확인
- "이 코드를 프로덕션에 넣어도 되는가?"의 최종 판단
