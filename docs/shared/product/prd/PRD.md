## 1. Background

AI를 업무에 도입한 팀이 늘고 있다. 하지만 대부분의 팀은 **개인이 AI를 각자 쓰는 상태**에 머물러 있다. AI가 팀의 협업 구조 안에 설계되어 있지 않다.

우리 팀(4인, PM/Eng/Design/UXR)은 이 문제를 직접 겪으며 리서치했다. 4명 전원 인터뷰(2026-03-18~25)에서 동일한 패턴이 확인됐다:

> **"결과물은 넘어가지만, 그걸 만든 사고과정은 사라진다."** (4/4 합의)

---

## 2. Problem

문제는 4개의 레이어로 쌓여 있다. **이 프로덕트(TeamBrain Assessment)는 Layer 1+2를 해결한다.** Layer 3+4는 별도 프로덕트(PRD-2)에서 다룬다.

### Layer 1. 게을러짐 — AI에게 사고까지 위임한다 ← **이 프로덕트가 해결**

AI 사용량이 많아질수록 인간은 위임하면 안 되는 것까지 위임하려 한다. "해줘"를 반복하면 판단력이 퇴화한다.

- **근거**: 4/4 팀원이 "핵심 역량이 전제되어야 AI가 의미있다"고 답변
- PM: "설계하는 팀과 안 하는 팀의 차이"
- Design: "원래의 디자인 규율이 전제"
- Eng: "일잘러가 AI 쓰면 더 잘하고, 아닌 사람은 여전히 평범"
- UXR: "AI 없이 리서치 할 줄 알아야 AI와 리서치를 할 수 있다"

### Layer 2. 사고의 룰이 없다 ← **이 프로덕트가 해결**

직무별로 어떤 사고 단계를 거쳐야 하는지, AI에게 무엇을 맡기고 무엇을 직접 해야 하는지에 대한 팀 합의가 없다.

- 같은 태스크를 누가 하느냐에 따라 AI 활용도와 결과물 품질이 크게 달라짐
- 어디까지 위임 가능한지에 대한 조직적 합의가 없어 개인의 판단에 의존

### Layer 3. Reasoning이 유실된다 ← PRD-2

다음 사람에게 넘기는 순간 "왜 이렇게 했는지"가 사라진다.

### Layer 4. 조직의 뇌를 볼 수 없다 ← PRD-2

팀 전체가 지금 무슨 생각을 하고 있는지, 누가 어떤 영역을 담당하는지 볼 수 없다.

## 3. Solution

> **30분 테스트로 나와 팀의 AI 협업 방식을 진단하고, 사고를 강제하는 룰을 자동 생성한다.**

| 입력 | 처리 | 출력 |
|------|------|------|
| 30분 테스트 답변 | AI가 분석 | 개인 AI 활용도 리포트 |
| 팀 전원의 리포트 | 교차 분석 | 팀 분석 리포트 + 협업 플로우 제안 |
| 리포트의 추천 | zip 다운로드 | {이름}.md + CLAUDE.md 생성 |

## 5. User Flow

대시보드 하나에서 전부. 채널 이동 없음.

```
┌─ 개인 (무료, 가입 없이) ─────────────────────────────┐
│                                                     │
│  1. 대시보드 접속 → 이메일 매직 링크 로그인             │
│  2. 이름, 역할, seniority 입력                        │
│  3. AI 리터러시 체크 (~25문항)                         │
│  4. 강제 선택 60쌍 (~15분)                            │
│  5. 완료 → 개인 리포트 즉시 생성                       │
│  6. zip 다운로드:                                     │
│     {이름}-report.md  (개인 리포트 풀버전)              │
│     {이름}.md         (Claude Code 사고 강제 룰)       │
│                                                     │
│  → .claude/ 폴더에 넣으면 바로 적용                    │
│  → 여기까지 혼자서도 가치 있음                         │
│                                                     │
└─────────────────────────────────────────────────────┘
                        ↓ 팀 만들기
┌─ 팀 ────────────────────────────────────────────────┐
│                                                     │
│  7. 오너가 "팀 만들기" → 팀원 이메일로 초대             │
│  8. 초대받은 팀원들이 각자 테스트 완료                   │
│     (개인 결과는 Supabase에 임시 저장)                  │
│  9. 전원 완료 → 팀 리포트 자동 생성                     │
│ 10. 오너에게 이메일로 team.zip 전송:                    │
│     team-report.md     (팀 리포트 풀버전)               │
│     CLAUDE.md          (팀 룰)                        │
│     {이름}.md × 전원   (팀 맥락으로 보강됨)             │
│                                                     │
│  → 오너가 .claude/ 폴더에 넣고 git push → 끝           │
│                                                     │
└─────────────────────────────────────────────────────┘
                        ↓
┌─ 일상 ──────────────────────────────────────────────┐
│                                                     │
│  VS Code에서 작업 시작                                │
│  → CLAUDE.md + {이름}.md 자동 로드                     │
│  → AI가 사고를 강제 (STOP/THINK 룰에 따라)             │
│                                                     │
└─────────────────────────────────────────────────────┘
```

### 배포 전략

GitHub 연동 없음. zip 다운로드 방식. 보안 걱정 없음.

```
개인 결과: 테스트 완료 즉시 zip 다운로드
팀 결과: 전원 완료 시 오너 이메일로 zip 전송
오너가 zip 풀어서 .claude/ 폴더에 넣고 git push → 끝
```

## 6. 테스트 설계

### 설계 원칙

테스트는 이 프로덕트의 핵심이다. MBTI(93문항), CliftonStrengths(177문항), DISC처럼 **돈 주고 하는 인적성검사급** 품질이어야 한다.

| 원칙 | 설명 | 레퍼런스 |
|------|------|---------|
| **강제 선택 쌍** | 둘 다 매력적인 문장 중 더 나에 가까운 쪽 선택. 정답이 없어 게이밍 방지 | CliftonStrengths, OPQ |
| **20초 타이머** | 문항당 20초 제한. 직관적 응답 유도, "이상적 자아" 대신 "실��� 자아" 대답 | CliftonStrengths |
| **매칭된 매력도** | 양쪽 선택지가 동등하게 매력적. 사회적 바람직성 편향 최소화 | OPQ, Thurstonian IRT |
| **직무별 분기** | PM/Eng/Design/UXR/Growth마다 다른 리터러시 문항 | 자체 설계 |
| **20분 집중** | MBTI(93문항/30분)과 유사한 밀도. ~88문항 | MBTI |
| **6축 측정** | AI 협업 4축 + 팀 협업 2축을 강제 선택 쌍으로 측정 | CliftonStrengths + DISC |

### AI 협업 유형 체계 — 4축 16타입

MBTI가 E/I, S/N, T/F, J/P 4축으로 16타입을 정의하듯이, TeamBrain은 AI와 일하는 방식의 4축을 정의한다. 각 축의 양쪽은 동등하게 유효하며 장단점이 있다.

| 축 | 극 | 설명 | 장점 | 약점 |
|----|-----|------|------|------|
| **판단 기점** | **S** Self-Directed | 내 판단을 먼저 세우고 AI 활용 | 방향이 뚜렷 | AI의 새로운 관점을 놓침 |
| | **A** AI-Directed | AI 제안을 받고 내가 선별/조정 | 넓은 탐색 | 본인 판단이 약해질 수 있음 |
| **협업 깊이** | **D** Directive | AI에게 명확히 시키고 결과 받음 | 효율적, 명확 | 깊이 부족 |
| | **G** Dialogic | AI와 주고받으며 같이 발전 | 깊이, 풍부 | 느림 |
| **AI 이해도** | **T** Strategic | AI 강점/약점 파악, 의도적 배치 | 예측 가능, 안정 | 새로운 활용 못 찾음 |
| | **E** Experimental | 새 활용법 시도, 경계를 넓힘 | 혁신적 | 실패 위험 |
| **품질 관여** | **C** Craftsperson | AI 결과를 내 기준으로 다듬고 검증 | 높은 품질 | 느림 |
| | **L** Leverager | AI 결과가 기준 이상이면 빠르게 활용 | 빠름 | 이슈 놓칠 수 있음 |

**16타입 예시:**

| 타입 | 이름 | 설명 |
|------|------|------|
| SDTC | Strategic Commander | 내가 판단→명확히 시킴→AI 파악→꼼꼼 검증. "AI는 내 전략의 실행 도구" |
| SGTC | Collaborative Architect | 내가 판단→AI와 토론→전략적→꼼꼼 검증. "비전이 있되 AI와 깊이 다듬는다" |
| AGEL | Creative Explorer | AI 제안→대화로 발전→실험적→빠르게 활용. "AI와 같이 탐험하며 예상 못한 걸 찾는다" |
| ADEL | Rapid Curator | AI 제안→시키고 결과→실험적→빠르게 활용. ⚠️ "무지성 해줘"에 가장 가까움 |
| SDEC | Adaptive Specialist | 내가 판단→시키고 결과→실험 시도→꼼꼼 검증. "기본은 내가 하되 새 방법도 시도" |
| AGTL | Flow Collaborator | AI 제안→대화로 발전→전략적→빠르게 활용. "AI와 호흡 맞춰 빠르게 결과" |

**팀 조합 분석:**

```
Robin(SGTC) + 우석(ADEL)
→ 축 1: S vs A — Robin은 방향을 먼저 잡고, 우석은 AI에서 출발 → 핸드오프에서 방향 갭
→ 축 4: C vs L — Robin은 다듬고, 우석은 빠르게 → 품질 기대치 차이
→ 추천: Robin→우석 핸드오프에 "방향의 이유"를 반드시 포함
        우석→Robin 핸드오프에 "검증한 부분/안 한 부분"을 명시
```

**측정 방법:**
```
축 1 (S/A) ← 강제 선택 10쌍
축 2 (D/G) ← 강제 선택 10쌍
축 3 (T/E) ← 강제 선택 10쌍 + AI 리터러시 경험 체크
축 4 (C/L) ← 강제 선택 10쌍
축 5 (E/I) ← 강제 선택 10쌍
축 6 (F/B) ← 강제 선택 10쌍
```

### Research Questions (내부 — 유저에게 보이지 않음)

테스트의 모든 질문은 아래 RQ 중 하나에 매핑된다.

| RQ | 알아내려는 것 | 리포트에서의 역할 |
|----|-------------|-----------------|
| **RQ1** | 이 사람은 누구인가? (역할, seniority, KPI, AI 리터러시, 강점/약점) | 개인 프로파일 |
| **RQ2** | 위임 경계는 어디인가? (AI에 맡기는 것 / 안 되는 것 / 맡기지만 위험한 것) | 위임 경계 분석 |
| **RQ3** | 품질 기준은 무엇인가? ("잘 했다"의 정의, 암묵적 감각) | 사고 프레임워크 |
| **RQ4** | 핸드오프에서 뭐가 유실되는가? (보낼 때 빠지는 것, 받을 때 없는 것) | 인풋/아웃풋 정의 |
| **RQ5** | 팀원 간 인식 충돌이 있는가? | 팀 리포트 충돌 분석 |

### 방법론 (내부)

CDM(Critical Decision Method) 기반. 강제 선택 쌍이 실제로는 인지 과업 분석을 수행한다.

**코딩 프레임워크** — 각 답변이 내부적으로 태깅됨:

| 코드 | 의미 | 리포트에서 |
|------|------|----------|
| `[DF]` | 자유 위임 | "AI에게 맡겨도 됩니다" |
| `[HR]` | 사람 필수 | "반드시 당신이 먼저 판단하세요" |
| `[DZ]` | 위험 위임 (맡기고 있지만 위험) | "주의: AI와 함께 하되 단계별로" |
| `[FB]` | 두려움 기반 비위임 | "시도해볼 만합니다" |
| `[WY]` | Why가 빠짐 | 핸드오프에 reasoning 필수 |
| `[CT]` | 맥락이 빠짐 | 핸드오프에 배경 정보 필수 |
| `[CF]` | 팀원 간 충돌 | 팀 논의 필요 |

체크 수 + 단계별 위치로 리터러시 레벨(Lv1~5) 산출.

전체 직무별 문항 → `docs/shared/product/specs/assessment-questions.md` Section B 참조.

---

#### Phase 3: 강제 선택 60쌍 (15분) — 유형 측정 핵심

타이머 없음. 5점 척도 (강하게 왼쪽 ~ 강하게 오른쪽).
둘 다 매력적인 문장 중 더 나에게 가까운 쪽 선택. 정답 없음.

축별 10쌍 × 6축 = 60쌍. 축 순서 섞어서 출제.

전체 60쌍 → `docs/shared/product/specs/assessment-questions.md` Section C 참조.

---

#### Phase 4: 확인 (1분)

유형 코드 + 리터러시 레벨 표시. 본인이 확인.

## 7. 개인 리포트

테스트 완료 후 생성. **MBTI급 수 페이지 분량. 비주얼 + 구체적 행동 추천.**

Claude API가 6축 점수 + AI 리터러시 + 역할을 받고 아래 구조로 생성.

### 7.1 유형 개요

```
┌──────────────────────────────────────────────────┐
│  Robin (변진영)                                    │
│  PM · Mid (5년차) · AI 리터러시 Lv4               │
│                                                   │
│  유형: SGTC-EF                                     │
│  "Collaborative Architect"                        │
│                                                   │
│  내가 먼저 판단하고(S), AI와 깊이 대화하며(G),      │
│  전략적으로 활용하고(T), 결과를 꼼꼼히 다듬는다(C).  │
│  동료에게 맥락을 명시적으로 전달하고(E),             │
│  갈등 시 근거를 들어 옹호한다(F).                   │
└──────────────────────────────────────────────────┘
```

- 핵심 특성 2-3 단락 서술
- 이 유형의 PM이 가장 빛나는 순간
- 이 유형의 PM이 가장 위험한 순간
- 강점 3가지 (구체적 업무 맥락)
- 약점/함정 3가지 (구체적 업무 맥락)

### 7.2 축별 심층 분석 (6축 각각)

각 축에 대해:
- 점수 + 스펙트럼 위치 비주얼
- "이 점수가 의미하는 것" — 추상적 설명이 아닌 구체적 행동 예시
- 업무에 미치는 영향 (PM이면 PM 업무 맥락에서)
- 같은 역할에서 반대 극의 사람과의 차이
- 주의할 것

예시 (축 1):
```
축 1: 판단 기점 — Self-Directed (72%)

당신은 AI에게 물어보기 전에 자기 생각을 먼저 정리하는 사람입니다.
PM으로서 이것은 강점입니다 — 제품 방향이 AI의 제안이 아닌 
당신의 전략적 판단에서 나온다는 뜻이기 때문입니다.

하지만 72%는 극단적이지 않습니다. 때때로 AI에서 출발하는 것도
받아들이는 유연성이 있습니다. 로드맵이나 경쟁 분석처럼 
데이터가 많은 업무에서는 AI가 먼저 옵션을 제시하게 하는 것도 
고려해보세요.

반대 극(AI-Directed)인 동료가 있다면: 그 사람은 AI에서 출발해서
넓게 탐색한 후 방향을 잡습니다. 당신의 방향성 + 그 사람의 탐색 범위를
결합하면 더 나은 결과를 낼 수 있습니다.
```

팀 전원 완료 시 자동 생성. **유형+직무 결합 분석. 모든 페어에 대해 구체적 추천.**

### 8.1 팀 유형 구성

```
┌─────────────────────────────────────────────────┐
│  팀 구성                                         │
│                                                  │
│  Robin  SGTC-EF  PM     Lv4  시스템 설계         │
│  건희   ADTC     Eng    Lv3  숙련               │
│  우석   AGEL-IB  Design Lv2  활용               │
│  아롬   SGTC-EF  UXR    Lv3  숙련               │
│                                                  │
│  축별 팀 분포 비주얼 (6축 각각에 4명의 위치)       │
└─────────────────────────────────────────────────┘
```

### 8.2 팀 강점 분석

유형 조합에서 오는 강점 (구체적 업무 맥락):
- "PM과 UXR이 모두 Self-Directed → 리서치와 전략이 한 방향으로 정렬되기 쉬움"
- "PM과 Eng이 둘 다 Craftsperson → 결과물 품질 기준이 높고 일관됨"
- "4명 전원 다른 유형 → 다양한 관점에서 의사결정"

### 8.3 팀 마찰 지점 — 모든 페어 분석

**모든 2인 조합**에 대해 구체적 분석:

```
Robin(SGTC-EF PM) ↔ 우석(AGEL-IB Design)

마찰 1 — 소통 갭:
  Robin은 Explicit(E), 우석은 Implicit(I)
  Robin이 상세한 PRD를 보내도 우석은 핵심만 읽음
  → 솔루션: Robin은 상단 3줄 요약 필수. 우석은 "왜 이 방향?" 한 줄 피드백 필수.

마찰 2 — 품질 기대 차이:
  Robin은 Craftsperson(C), 우석은 Leverager(L)
  Robin은 결과물을 95%까지 다듬으려 하고, 우석은 80%면 넘김
  → 솔루션: "이 작업의 품질 기준은 X다"를 작업 시작 전에 합의.

마찰 3 — 갈등 스타일:
  Robin은 Firm(F), 우석은 Bridging(B)
  의견 충돌 시 Robin이 주도하고 우석이 양보할 가능성
  → 솔루션: Robin은 우석에게 피드백 줄 때 "질문 형태"로.
           "이거 고쳐야 해" 대신 "이 부분 어떻게 생각해?"
```

```
Robin(SGTC-EF PM) ↔ 건희(ADTC Eng)

궁합 좋은 점:
  둘 다 Craftsperson(C) → 품질 기준 일치
  Robin은 Explicit(E) → 건희가 원하는 상세 문서를 자연스럽게 제공

마찰 — 판단 기점 차이:
  Robin은 Self-Directed(S), 건희는 AI-Directed(A)
  Robin은 방향을 먼저 잡지만, 건희는 AI 옵션에서 출발
  → 건희가 Robin의 방향과 다른 AI 옵션을 발견하면 충돌 가능
  → 솔루션: Robin이 넘길 때 "방향의 이유"와 "검토하지 않은 대안"을 명시.
           건희가 다른 옵션을 발견하면 "이런 옵션도 있는데 어떻게 생각해?"로 공유.
```

(모든 조합에 대해 동일 수준의 분석)

### 8.4 협업 파이프라인 — 우리 팀 유형에 맞춤

```
Step 1: PM(SGTC-EF) — 문제 정의
  🔴 STOP: Robin이 먼저 판단
  Robin은 S(먼저 판단) + G(AI와 대화) → 방향 잡고 AI와 검증

Step 2: PM → Design 전달
  Robin(E) → 우석(I): 3줄 요약 필수. 상세는 하단에.
  필수: 전략 의도 + 기각한 방향 + 우석이 확인할 것

Step 3: Design(AGEL-IB) — 옵션 탐색
  우석은 A(AI에서 출발) + G(대화) → AI에게 다양한 방향 뽑게 함
  우석은 L(빠르게 활용) → 빠르게 3-4개 옵션 생성

Step 4: Design — 방향 결정
  🔴 STOP: 우석이 먼저 판단
  "유저가 생각 안 해도 되는 상태인지" 확인

Step 5: Design → Eng 전달
  우석(I) → 건희(C): 우석이 이유를 안 쓸 가능성 높음
  필수: 선택 이유 3줄 + 기각 옵션 1개 + 해석 여지 명시

Step 6: Eng(ADTC) — 구현
  건희는 A(AI에서 출발) + T(전략적) → AI에게 코드 맡기되 검증
  건희는 C(꼼꼼) → 라인별 리뷰

Step 7: Eng — 코드 완성
  🟡 THINK: 가장 위험한 부분 + 테스트 커버리지 + 읽을 수 있는지
```

### 8.5 역할별 인풋/아웃풋 — 유형 기반

```
PM (SGTC-EF Robin)
  📥 받을 때 원하는 것: reasoning (Explicit 성향)
  📤 줄 때 잘하는 것: 상세 문서 (Explicit)
  📤 줄 때 주의할 것: 너무 긺 (Craftsperson 함정)

Design (AGEL-IB 우석)  
  📥 받을 때 원하는 것: 핵심만 짧게 (Implicit 성향)
  📤 줄 때 주의할 것: reasoning 누락 (Implicit + Leverager)
  📤 강제: 선택 이유 3줄 필수

Eng (ADTC 건희)
  📥 받을 때 원하는 것: 기술 제약 + 엣지케이스 (Craftsperson)
  📤 줄 때 잘하는 것: 꼼꼼한 기술 문서

UXR (SGTC-EF 아롬)
  📥 받을 때 원하는 것: 핵심 질문 3개 + 데드라인 (효율적 브리프)
  📤 줄 때 주의할 것: 리포트 과다 (같은 SGTC → Craftsperson 함정)
```

### 8.6 팀 사각지대

- "우석(Leverager)과 나머지(Craftsperson) 사이 품질 기대 갭 → 작업 시작 전 품질 기준 합의 필수"
- "AI-Directed(건희, 우석)의 reasoning이 결과물에 빠질 위험 → 결과물 전달 시 reasoning 3줄 강제"
- "UXR↔Eng 직접 전달 경로 없음 → 리서치 인사이트가 PM을 거쳐야 함. PM이 필터링/왜곡 가능 → 중요한 리서치는 아롬이 건희에게 직접 3줄 요약 전달하는 루트 추가"
- "팀 전원 Firm(F) 또는 Bridging(B) 한쪽에 쏠려있다면 의사결정 패턴 경고"

### 8.7 추천 팀 룰

팀 유형 조합에서 자연스럽게 도출:
1. "결과물 전달 시 'reasoning 3줄'을 필수로 포함한다" (Implicit 멤버의 reasoning 누락 방지)
2. "최종 판단은 반드시 사람이 먼저 한다" (전원 합의)
3. "작업 시작 전 '이 작업의 품질 기준'을 한 문장으로 합의한다" (C/L 갭 방지)
4. "피드백은 질문 형태로 한다" (F/B 갭 완화)

## 10. System Architecture

```
┌─ 대시보드 (Next.js / Vercel) ──────────────────┐
│                                                │
│  테스트 UI                                     │
│    ↓                                           │
│  Claude API로 리포트 생성                       │
│    ↓                                           │
│  개인: zip 다운로드                              │
│  팀: Supabase 임시 저장 → 전원 완료 시           │
│       오너에게 zip 이메일 (Resend)               │
│                                                │
└────────────────────┬───────────────────────────┘
                     │
              Supabase (임시 저장)
              - 팀 테스트 결과 (7일 후 삭제)
              - 팀 초대 상태
```

| 구성 요소 | 기술 | 비용 |
|-----------|------|------|
| 대시보드 | Next.js (Vercel 무료 배포) | $0 |
| 데이터 저장 | GitHub 레포 (teambrain 전용) | $0 |
| 분석 엔진 | Claude API (서버사이드, 우리 키) | 1인당 ~$0.09 |
| 임시 저장 | Supabase (팀 테스트 결과 임시 보관, 7일 삭제) | 무료 티어 |
| 인증 | Supabase Auth 매직 링크 (이메일만, 비번 없음) | 무료 티어 |
| 이메일 | Resend (팀 초대 + zip 전송, 월 100건) | 무료 티어 |
| zip 생성 | JSZip (서버 메모리에서 생성) | $0 |
| 파일 전달 | 개인: 브라우저 다운로드 / 팀: 이메일 첨부 | $0 |

**무료 서비스.** GitHub 연동 없음. 보안 걱정 없음.
- 개인: 테스트 → 리포트 → zip 다운로드

- 오너가 zip 풀어서 .claude/ 폴더에 넣고 git push → 끝
- Supabase 임시 데이터는 7일 후 자동 삭제

## 11. 오픈소스 도입 경험

```
개인:
1. teambrain.app 접속 (가입 없음)
2. 이름, 역할, seniority 입력
3. AI 리터러시 체크 (3분)
4. 강제 선택 60쌍 (15분)
5. 개인 리포트 확인
6. zip 다운로드 ({이름}-report.md + {이름}.md)
7. .claude/ 폴더에 넣기 → 끝

팀:
8. "팀 만들기" → 팀원 이메일로 초대
9. 팀원들이 각자 테스트 (각 20분)
10. 전원 완료 → 오너에게 team.zip 이메일
    (team-report.md + CLAUDE.md + 전원 {이름}.md)
11. 오너가 .claude/ 에 넣고 git push → 끝
```

## 12. Milestones

### M0: 프로토타입
- 테스트 UI (60쌍 + 리터러시)
- 개인 리포트 (Claude API 생성)
- 팀 리포트 (가상 데이터)
- {이름}.md + CLAUDE.md 생성
- zip 다운로드

### M1: 내부 팀 사용 (2주)
- 4명 전원 테스트 완료
- 실제 리포트 생성 + {이름}.md 적용
- 2주 사용 후 효과 측정

### M2: 공개 (4주)
- 대시보드 배포 (teambrain.app)
- 팀 초대 + Supabase 임시 저장 + 이메일 zip 전송
- 오픈소스: GitHub 퍼블릭

## 14. Risks & Open Questions

| 리스크 | 완화 |
|--------|------|
| 강제 선택 60쌍이 지루할 수 있음 | 진행률 바 + 축 순서 섞어서 단조로움 방지 |
| 리포트가 기대 이하면 신뢰 하락 | Claude API 프롬프트 퀄리티 + MBTI급 수 페이지 분량 |
| {이름}.md가 너무 빡빡하면 우회 | 사용 2주 후 재조정 권유 |
| Supabase 임시 데이터 보안 | 7일 자동 삭제 + 테스트 결과만 저장 (개인정보 최소) |
| 이메일 전송 실패 | Resend 재시도 + 대시보드에서 직접 다운로드 가능 |
